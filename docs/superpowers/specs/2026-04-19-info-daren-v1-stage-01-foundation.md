# 信息达人 V1 阶段文档 01：产品基础与系统骨架

## 文档信息

- 项目名称：信息达人
- 阶段主题：V1 产品定位、范围界定、系统原则、领域模型
- 当前状态：已在对话中确认
- 更新时间：2026-04-19

## 1. 当前产品目标

`信息达人 V1` 的目标不是做一个简单的信息聚合器，而是做一个面向大众热点用户的 `事件理解器`。

核心能力不是“把不同平台的内容堆在一起”，而是：

- 把同一热点在不同平台的内容聚成一个事件
- 给用户一条 `一句话看懂`
- 用 `事件时间线` 帮用户快速理解事情的发展过程
- 用简洁的背景补充与来源归纳帮助用户少切平台、更快看懂

## 2. 用户与场景定位

### 2.1 目标用户

- 核心用户：大众热点用户
- 用户诉求：想刷热点，但不想在多个平台之间来回切换
- 用户预期：用更低成本快速看懂今天正在发生的重要事情

### 2.2 核心场景

- 优先场景：刷热点但不想切平台
- 用户行为特征：
  - 打开应用后先快速浏览全网重要事件
  - 看到感兴趣的热点后点进详情
  - 希望先知道“这到底是什么”，再决定要不要深看

### 2.3 关键交互偏好

- 首页优先强调：`一句话看懂`
- 详情页优先强调：`事件时间线`
- 排序策略优先：`热度 + 时效混合`
- 首页信息架构：`综合流 + 分类入口`

## 3. 产品路线选择

在备选路线中，最终选择的是：

`以智能热榜型为主，吸收一部分多源解读型能力`

原因如下：

- 对大众用户最直接，学习成本低
- 能最大化承接“刷热点但不想切平台”的场景
- 首页负责“快”，详情页负责“懂”
- 既保留热点流的刷感，也能体现事件理解器的差异化价值

## 4. 产品定义与核心承诺

### 4.1 一句话产品定义

`信息达人是一个面向大众用户的事件理解器，用事件聚合替代平台切换，用一句话摘要和时间线帮助用户更快看懂全网热点。`

### 4.2 三个核心承诺

- `不切平台`：把多个平台内容聚成一个事件
- `不费脑子`：先给一句话结论
- `不怕跟不上`：用时间线告诉用户事情怎么发展到现在

## 5. V1 范围与非目标

### 5.1 V1 必做能力

- 事件聚合
- 一句话摘要
- 时间线整理
- 多来源归纳
- 背景补充
- 分类浏览
- 热度 + 时效排序

### 5.2 V1 暂不纳入

- 强个性化推荐
- 复杂订阅与提醒体系
- 用户评论社区
- 深度研报式分析
- 一次性覆盖所有终端形态

### 5.3 范围控制原则

V1 可以收敛，但不能为了收敛而牺牲未来扩展性。当前设计遵循：

- 功能从重点做起
- 架构按全面产品的方向设计
- 前期先服务好大众热点用户
- 后续逐步扩展推荐、搜索、订阅、运营和多端能力

## 6. 系统设计原则

为了保证产品未来能够成长为一个全面的平台，系统设计从一开始就遵循以下原则：

### 6.1 采集层和理解层分离

- 爬虫负责采集
- 事件聚类、摘要、时间线、背景补充等能力独立成“理解处理层”
- 避免把智能处理逻辑写死在爬虫内部

### 6.2 以事件为核心，而不是以原始信息为核心

- 原始内容是证据材料
- 事件才是用户真正消费的对象
- 前端和服务接口最终都应围绕 `event` 展开

### 6.3 处理链可插拔

以下能力应设计为独立处理器：

- 聚类
- 摘要生成
- 时间线提取
- 背景补充
- 热度计算

这样后续升级算法、替换模型、加入人工校验时，不会冲击整体架构。

### 6.4 前台体验和后台能力解耦

- 前端面向 `事件 API`
- 后端底层采集与清洗逻辑可持续演进
- Web/H5/小程序/App 后续都能复用统一事件服务

### 6.5 单租户 demo 也按平台思路设计

即使当前阶段是 demo，也建议从边界上按平台拆层：

- source ingestion
- normalization
- event intelligence
- serving API
- client app

## 7. 目标分层架构

### 7.1 数据采集层

负责：

- 爬虫采集

### 7.2 数据标准化层

### 7.3 事件理解层

负责：

- 时间线整理
- 背景补充
- 热度分计算

### 7.4 服务输出层

负责：

- 事件流 API
- 事件详情 API
- 分类页 API

### 7.5 应用交互层

负责：

- Web / H5
- 后续扩展到小程序 / App

## 8. 核心领域模型设计

当前建议采用“两层模型”：

- 底层：内容证据层
- 上层：事件产品层

### 8.1 内容证据层

用于承接采集数据、保留可追溯性、支持重算。

#### category

- 内容分类
- 例如：科技、财经、体育、国际

#### channel

- 内容来源渠道
- 例如：微博、路透、36氪、东方财富

#### raw_item

- 原始抓取内容
- 尽量保留来源标题、原始摘要、原始时间、原始 URL、抓取时间、抓取状态等字段

#### normalized_item

- 标准化后的内容单元
- 将不同平台字段对齐成统一结构
- 作为后续聚类与事件理解的直接输入

### 8.2 事件产品层

这是前端最终面向用户消费的主对象。

#### event

一个热点事件本体，例如“OpenAI 发布新模型能力”。

重点字段建议包括：

- `title`
- `one_line_summary`
- `status`
- `heat_score`
- `freshness_score`
- `composite_score`
- `primary_category_id`
- `started_at`
- `last_updated_at`

#### event_item_link

用于关联事件和证据内容，表示哪些 `normalized_item` 属于这个事件。

重点字段建议包括：

- `event_id`
- `item_id`
- `role`

其中 `role` 可区分：

- 首发来源
- 媒体解读
- 社交讨论
- 背景材料

#### event_timeline_entry

用于表示事件时间线节点。

重点字段建议包括：

- `event_id`
- `occurred_at`
- `summary`
- `source_item_id`
- `confidence`

#### event_summary_snapshot

用于保存事件摘要快照，支持后续优化和版本演进。

重点字段建议包括：

- `event_id`
- `summary_type`
- `content`
- `version`

建议先支持这些 `summary_type`：

- `one_line`
- `what_happened`
- `why_it_matters`
- `latest_update`

#### event_tag

用于标注事件标签，例如：

- AI
- 大模型
- 黄金
- 突发
- 财报

该模型对搜索、推荐、订阅都很重要。

## 9. 迁移思路

不建议一次性推翻当前 `info_aggregation` 的现有结构，而建议平滑演进：

1. 继续保留现有 `category / channel / info`
2. 将 `info` 逐步视作 `raw_item` 或 `normalized_item` 的前身
3. 新增 `event` 相关表
4. 新 API 优先面向 `event`
5. 老 API 暂时保留，保证前后端平滑迁移

## 10. 当前阶段结论

这一阶段已经确定了 `信息达人 V1` 的底层产品方向：

- 它不是简单的热榜搬运工具
- 它的核心对象是“事件”，不是“帖子”或“新闻条目”
- 它的价值是帮助用户在更短时间内理解更复杂的热点
- 它需要从现在的采集 demo，演进成一个具备事件理解能力的平台
