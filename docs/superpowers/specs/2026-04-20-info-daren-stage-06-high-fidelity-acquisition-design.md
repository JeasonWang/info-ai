# 信息达人 V1 Stage 06 高完整度采集框架设计

- 日期：`2026-04-20`
- 阶段：`V1 / 数据采集能力增强`
- 当前状态：`设计已确认，待用户复核后进入 implementation plan`

---

## 1. 目标

当前系统已经具备热点事件流 MVP 闭环，但底层采集能力仍偏向“轻量热点抓取器”，无法稳定保证正文完整性与详情准确性。现阶段的主要问题包括：

- 列表页摘要直接入库，信息天然残缺
- 详情抓取策略单一，易被反爬、壳页或页面结构变化影响
- 成功判定过于宽松，抓到少量文本也会被视为详情成功
- 失败时保留列表摘要，导致系统表面“有内容”但真实性和完整性不足

本阶段目标是把现有链路升级为 `高完整度采集框架`，先在微博渠道落地验证，再为其他渠道复用打基础。

一句话定义：

`系统不再以“拿到一点文本”为成功，而是以“拿到经过校验的可信正文”为成功。`

---

## 2. 范围

### 2.1 本阶段纳入

- 抽象通用高完整度采集框架
- 将列表抓取与详情解析职责明确分离
- 为详情抓取引入多策略解析与降级回退
- 增加正文有效性校验与完整度评分
- 为每条 `Info` 记录保存详情抓取质量状态
- 优先在 `微博` 渠道接入并验证

### 2.2 本阶段暂不纳入

- 扩充新的渠道来源
- 做浏览器自动化或重型代理池系统
- 做全链路可视化监控后台
- 对所有现有爬虫一次性全面重写

本阶段强调：

`先把框架抽对，再用微博把“高完整度”能力做实。`

---

## 3. 总体架构

当前链路为：

`crawl -> clean -> save -> fetch_detail -> rebuild_events`

本阶段升级后，详情采集链路调整为：

`ListFetcher -> DetailResolver -> ContentNormalizer -> ContentValidator -> CompletenessScorer -> AcquisitionEvidence -> save/update -> rebuild_events`

### 3.1 ListFetcher

负责抓取热点列表，只输出候选事件条目与最基础上下文，不对“详情完整性”作任何承诺。

职责：

- 抓取榜单或列表页候选项
- 产出 `title / source_url / source_id / event_time` 等基础字段
- 尽量保留列表来源提供的补充字段，供详情解析使用

### 3.2 DetailResolver

负责详情获取与多策略回退，是本阶段的核心增强点。

职责：

- 定义渠道级详情策略链
- 顺序执行多个详情策略
- 对每次策略结果进行初步归并
- 在某个策略拿到高可信正文后提前结束
- 当高优先策略失败时自动回退到低优先策略

### 3.3 ContentNormalizer

对抓取结果进行清洗与归一化，避免把 HTML 壳文本、重复句、平台噪声直接入库。

职责：

- 去 HTML 标签、脚本、样式、噪声导航文案
- 去重和合并多段正文
- 清洗空白字符、异常短句、平台冗余词
- 输出适合后续校验与评分的标准文本

### 3.4 ContentValidator

负责判断“抓到的到底是不是可信正文”。

职责：

- 识别反爬提示页、备案页、壳页面、跳转页
- 识别过短文本或与主题弱相关的文本
- 区分列表摘要、部分详情与高可信正文
- 对不合格内容明确返回失败原因

### 3.5 CompletenessScorer

负责为详情结果打完整度分，避免只有二元成功/失败。

评分维度包括：

- 正文长度
- 信息密度
- 与标题/主题的相关性
- 来源证据数量
- 是否命中无效内容规则
- 是否为策略链中的高优先来源

### 3.6 AcquisitionEvidence

负责记录采集证据和执行结果，为后续调优与分析提供依据。

内容包括：

- 本次使用的详情策略
- 正文长度
- 完整度评分
- 判定状态
- 命中的无效规则
- 失败原因
- 执行时间

## 4. 微博落地设计

微博是本阶段首个重点落地渠道。当前微博实现的问题在于：

- 热榜列表能抓到标题，但正文主要来自搜索结果拼接
- 详情策略少，容易被接口波动和反爬影响
- 成功标准过松，短文本也会被当作有效详情

因此微博渠道改造为 `多策略详情解析器`。

### 4.1 微博详情策略顺序

推荐执行顺序如下：

1. `topic_search`
   - 通过话题关键词搜索微博相关结果
   - 提取前几条高相关微博正文
   - 组合为多证据正文

2. `hot_band_context`
   - 使用热榜接口已有的 `note / raw_text / desc` 等字段补全背景
   - 作为正文增强，而不是单独视为完整详情

3. `mobile_search`
   - 从移动端搜索容器页中提取 `mblog.text`
   - 对多条结果进行清洗与拼接

4. `web_fallback`
   - 网页搜索或页面文本提取兜底
   - 仅视为低可信结果，不能和前几种策略同权

### 4.2 微博正文生成原则

微博详情不再依赖“某一个接口恰好返回一段文本”，而是采用：

`多条相关微博证据 -> 清洗 -> 去重 -> 合并 -> 校验 -> 评分`

这意味着最终正文更接近“事件证据摘要”，而不是单条微博复制。

### 4.3 微博有效性校验

微博渠道需要重点增加以下规则：

- 命中反爬提示、登录引导、页面异常文案时判失败
- 命中导航、壳页面、通用搜索页噪声时判失败
- 文本过短、重复率过高时判为 `partial` 或 `failed`
- 与标题核心词弱相关时不得判为 `complete`

## 5. 数据模型与状态设计

为了让系统真正感知“采集完整性”，需要在 `Info` 上增强详情质量状态，而不是只有当前的粗粒度成功/失败。

### 5.1 Info 质量字段

在现有 `Info` 模型基础上，扩展以下字段：

- `detail_fetch_status`
  - `pending`
  - `list_only`
  - `partial`
  - `complete`
  - `failed`

- `detail_fetch_error`
  - 保存规范化失败原因，例如：
  - `anti_crawl_blocked`
  - `shell_page`
  - `content_too_short`
  - `invalid_payload`
  - `network_timeout`

- `detail_strategy`
  - 标识当前最终采用的详情策略，如 `topic_search`

- `detail_score`
  - 详情完整度得分

- `detail_content_length`
  - 归一化正文长度

- `detail_fetched_at`
  - 最近一次详情抓取完成时间

### 5.2 采集日志表

为后续数据分析与策略调优，建议新增采集日志表，例如 `InfoAcquisitionLog`。

建议记录：

- `info_id`
- `channel_code`
- `strategy`
- `status`
- `score`
- `content_length`
- `failure_reason`
- `matched_rules`
- `raw_excerpt`
- `created_at`

该表用于回答类似问题：

- 哪个策略成功率最高
- 哪个策略最容易被反爬拦截
- 哪个渠道最常出现壳页面
- 哪种失败原因占比最高

---

## 6. 执行策略

本阶段建议分两层推进：

### 6.1 第一层：框架先落地

先抽出采集框架公共能力，包括：

- 详情策略执行器
- 内容归一化器
- 内容校验器
- 完整度评分器
- 采集证据记录器

目标是让后续每个渠道都能复用相同的成功判定和质量记录方式。

### 6.2 第二层：微博优先接入

在框架成型后，先把微博接入为首个高完整度渠道。

目标包括：

- 多策略详情解析生效
- 详情状态从 `success/failed` 升级为质量分层
- 微博详情内容比当前版本更长、更相关、更少错抓

---

## 7. 验证方式

本阶段不应只验证“接口还能跑”，而应验证“完整度确实提升”。

验证方式包括：

- 单元测试
  - 校验内容过滤、状态映射、评分逻辑

- 渠道测试
  - 用微博样例验证不同策略的返回结果

- 回归测试
  - 确保现有保存、详情更新、事件重建链路不被破坏

- 质量验证
  - 对比改造前后的：
  - 有效详情率
  - 平均详情长度
  - 失败原因分布
  - 列表摘要误判为详情的比例

---

## 8. 风险与边界

本阶段的主要风险：

- 微博接口和页面结构持续变化
- 无浏览器自动化时，仍可能受登录态或反爬限制
- 不同渠道的“完整正文”定义并不完全一致

因此本阶段的边界是：

`我们先做到“高可信详情采集框架 + 微博显著增强”，而不是承诺所有渠道、所有页面都能完全抓满。`

这也是当前最稳妥、最可持续的技术路线。

---

## 9. 结论

下一阶段的核心不是扩来源，而是升级采集底座。

最终方向明确为：

- 抽象通用高完整度采集框架
- 强化详情校验、评分与证据记录
- 先把微博做成首个高完整度样板渠道

这样后续再扩展到头条、小红书、知乎、财经媒体时，系统会建立在稳定、可分析、可调优的采集能力之上，而不是持续堆叠脆弱的渠道特例代码。
