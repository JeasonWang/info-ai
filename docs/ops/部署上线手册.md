# 信息达人部署上线手册

> 适用版本：v1.0.0 AI 情报台上线候选版  
> 更新时间：2026-05-15  
> 当前部署方式：Docker Compose 启动 Redis 和四个应用容器，MySQL 使用宿主机或独立数据库

## 1. 部署对象

当前生产部署包含：

| 服务 | 容器名 | 镜像 | 对外访问 |
|---|---|---|---|
| Redis | `info-redis` | `redis:7-alpine` | 仅服务器本机 `127.0.0.1:16379` |
| 采集服务 | `info-aggregation` | `info-ai-info-aggregation:pro` | 仅服务器本机 `127.0.0.1:18000` |
| 业务 API | `info-serve` | `info-ai-info-serve:pro` | `服务器IP:8085` |
| 用户端 | `info-mvp` | `info-ai-info-mvp:pro` | `服务器IP:8082` |
| 管理后台 | `info-admin` | `info-ai-info-admin:pro` | `服务器IP:8081` |

生产建议：

- 公网只开放 `80/443`，由 Nginx、Caddy 或云网关转发到 `info-mvp`、`info-admin` 和 `info-serve`。
- 不公网开放 MySQL。
- 不公网开放 `info_aggregation`。
- 渠道 Cookie / ZSE 通过管理后台凭证管理维护，不写入 `.env`。

## 2. 服务器准备

服务器建议：

- Ubuntu 22.04/24.04 或等价 Linux。
- Docker Engine 和 Docker Compose v2。
- MySQL 8.x，运行在宿主机 `3306` 或独立数据库。
- 生产目录建议为 `/opt/info-ai`。

```bash
sudo mkdir -p /opt/info-ai
sudo chown -R "$USER:$USER" /opt/info-ai

docker version
docker compose version
mysql --version
```

## 3. 数据库完整初始化记录

### 3.1 当前数据库事实来源

当前数据库名称仍为 `info-max`。这是历史库名，不代表当前用户端仍使用 `info-max`。

| 文件 | 职责 |
|---|---|
| `info_aggregation/sql/mysql_schema_pro.sql` | 创建数据库和完整表结构 |
| `info_aggregation/sql/mysql_migration_max.sql` | 初始化基础分类、渠道、采集任务、默认管理员、质量快照、默认 LLM 配置 |
| `info_aggregation/sql/migration_v1.1.0_channel_credentials.sql` | 旧库升级：渠道凭证数据库化 |
| `info_aggregation/sql/migration_v1.1.0_event_analysis_source.sql` | 旧库升级：事件分析来源追溯 |
| `info_aggregation/sql/migration_v1.2.0_event_evolution.sql` | 旧库升级：事件演变和历史脉络 |
| `info_aggregation/sql/migration_v1.3.0_database_cleanup.sql` | 旧库升级：清理废弃表和冗余字段 |
| `info_aggregation/sql/migration_v1.4.0_incremental_rebuild.sql` | 旧库升级：增量事件重建检查点 |
| `info_aggregation/sql/migration_v1.5.0_event_display_quality.sql` | 旧库升级：事件展示质量字段 |

### 3.2 全新数据库初始化

首次部署全新库时执行：

```bash
cd /opt/info-ai

mysql -h 127.0.0.1 -uroot -p < info_aggregation/sql/mysql_schema_pro.sql
mysql -h 127.0.0.1 -uroot -p < info_aggregation/sql/mysql_migration_max.sql
```

初始化完成后检查：

```bash
mysql -h 127.0.0.1 -uroot -p -D info-max -e "
SHOW TABLES;
SELECT COUNT(*) AS category_count FROM category;
SELECT COUNT(*) AS channel_count FROM channel;
SELECT COUNT(*) AS crawl_task_count FROM crawl_task;
SELECT email, role, status FROM user_account WHERE role = 'admin';
"
```

全新库应至少具备：

- 27 张核心表。
- 6 个基础分类。
- 12 个初始化渠道。
- 每个渠道一条 `crawl_task`。
- 默认管理员账号 `admin@info-daren.local`。
- 默认 LLM 配置 `qwen` 和 `deepseek`，默认关闭。

### 3.3 默认管理员处理

`mysql_migration_max.sql` 会初始化：

```text
admin@info-daren.local / Admin123456
```

生产环境必须在首次登录后立即修改密码，或使用 `create-admin` 工具覆盖：

```bash
cd /opt/info-ai/info-serve

INFO_SERVE_MYSQL_DSN='root:你的MySQL密码@tcp(127.0.0.1:3306)/info-max?charset=utf8mb4&parseTime=true&loc=Local' \
go run ./cmd/create-admin -email admin@info-daren.local -password '替换为强密码'
```

注意：当前 `deploy.sh` 会在首次生成 `.env` 时写入 `INFO_ADMIN_EMAIL` 和 `INFO_ADMIN_PASSWORD`，但 `info-serve` 服务启动不会自动读取这两个变量覆盖数据库管理员密码。生产环境不要只依赖 `.env` 里的管理员密码，必须确认数据库中的管理员账号已经更新。

### 3.4 已有数据库升级

已有生产库升级前必须备份：

```bash
mysqldump -h 127.0.0.1 -uroot -p info-max > backup-info-max-$(date +%Y%m%d%H%M%S).sql
```

再按顺序执行增量迁移：

```bash
cd /opt/info-ai

mysql -h 127.0.0.1 -uroot -p info-max < info_aggregation/sql/migration_v1.1.0_channel_credentials.sql
mysql -h 127.0.0.1 -uroot -p info-max < info_aggregation/sql/migration_v1.1.0_event_analysis_source.sql
mysql -h 127.0.0.1 -uroot -p info-max < info_aggregation/sql/migration_v1.2.0_event_evolution.sql
mysql -h 127.0.0.1 -uroot -p info-max < info_aggregation/sql/migration_v1.3.0_database_cleanup.sql
mysql -h 127.0.0.1 -uroot -p info-max < info_aggregation/sql/migration_v1.4.0_incremental_rebuild.sql
mysql -h 127.0.0.1 -uroot -p info-max < info_aggregation/sql/migration_v1.5.0_event_display_quality.sql
```

检查关键字段：

```bash
mysql -h 127.0.0.1 -uroot -p -D info-max -e "
SHOW COLUMNS FROM channel LIKE 'cookies';
SHOW COLUMNS FROM event LIKE 'display_quality_score';
SHOW TABLES LIKE 'event_analysis_source';
SHOW TABLES LIKE 'rebuild_checkpoint';
"
```

## 4. `.env` 配置

生产服务器在 `/opt/info-ai/.env` 保存配置。不要提交真实 `.env`。

最小必填项：

```text
DB_TYPE=mysql
DB_HOST=host.docker.internal
DB_PORT=3306
DB_USER=root
DB_PASSWORD=你的MySQL密码
DB_NAME=info-max

INFO_SERVE_MYSQL_DSN=root:你的MySQL密码@tcp(host.docker.internal:3306)/info-max?charset=utf8mb4&parseTime=true&loc=Local
INFO_SERVE_SESSION_SECRET=替换为强随机字符串

REDIS_PASSWORD=
REDIS_DB=0
ENABLE_REDIS_COMMAND_CONSUMER=1

ENABLE_PUBLIC_API=1
EVENT_ANALYSIS_ENABLE_LLM=0
EVENT_ANALYSIS_LLM_RETRY_TIMES=2
EVENT_ANALYSIS_LLM_FAILURE_THRESHOLD=3
EVENT_ANALYSIS_LLM_COOLDOWN_MINUTES=30

VITE_API_BASE_URL=/api

PUBLIC_SITE_URL=http://服务器IP:8082
PUBLIC_ADMIN_URL=http://服务器IP:8081
PUBLIC_API_URL=http://服务器IP:8085
PUBLIC_AGGREGATION_URL=http://127.0.0.1:18000
```

注意：

- `docker-compose.yml` 不启动 MySQL 容器，`DB_PASSWORD` 必须和宿主机或独立数据库一致。
- `host.docker.internal` 通过 compose 的 `extra_hosts` 映射到宿主机网关。
- `EVENT_ANALYSIS_ENABLE_LLM=0` 表示默认使用规则分析，后台可配置模型后再启用。
- 渠道凭证不再放 `.env`，统一在管理后台凭证管理维护。

## 5. 本机打包部署

适用于直接在服务器上拉取代码并构建镜像。

```bash
cd /opt/info-ai
chmod +x ./deploy.sh
./deploy.sh prod local-build
```

脚本流程：

1. 检查 Docker 和 Compose。
2. 如果 `.env` 不存在，自动生成一份模板。
3. 执行 `docker compose -f docker-compose.yml build`。
4. 执行 `docker compose -f docker-compose.yml up -d --remove-orphans --no-build`。
5. 检查 `info-aggregation`、`info-serve`、`info-mvp`、`info-admin` 容器是否运行。

首次部署注意：

- 如果脚本自动生成了 `.env`，需要先确认 MySQL 密码是否正确。
- 脚本不会自动执行数据库初始化或迁移。
- 脚本不会自动修改数据库管理员密码。

## 6. GitHub Actions 部署

当前 workflow 位于 `.github/workflows/deploy.yml`。

触发方式：

- 推送到 `main` 或 `master`。
- 在 GitHub Actions 页面手动运行 `Deploy`。

需要配置的 GitHub Secrets：

| Secret | 说明 |
|---|---|
| `SERVER_HOST` | 服务器公网 IP 或域名 |
| `SERVER_USER` | SSH 用户 |
| `SERVER_SSH_KEY` | SSH 私钥 |
| `SERVER_DEPLOY_PATH` | 部署目录，推荐 `/opt/info-ai` |

可选 GitHub Variable：

| Variable | 默认值 | 说明 |
|---|---|---|
| `VITE_API_BASE_URL` | `/api` | 前端 API 基地址 |

CI 流程：

1. 校验 `docker-compose.yml`、`deploy.sh`、基础 SQL 存在。
2. 构建四个应用镜像。
3. 导出 `images/info-ai-images.tar.gz`。
4. 上传镜像包、`docker-compose.yml`、`deploy.sh`、基础 SQL 到服务器。
5. 在服务器执行 `./deploy.sh prod image-tar`。

当前限制：

- workflow 当前只上传 `mysql_schema_pro.sql` 和 `mysql_migration_max.sql`。
- 如果是旧库升级，需要人工把 `info_aggregation/sql/migration_*.sql` 同步到服务器后再执行迁移。
- workflow 不执行数据库初始化和迁移。

## 7. 使用镜像包部署

适用于 GitHub Actions 已经上传 `images/info-ai-images.tar.gz` 的服务器。

```bash
cd /opt/info-ai
chmod +x ./deploy.sh
./deploy.sh prod image-tar
```

脚本会执行：

```bash
gzip -dc images/info-ai-images.tar.gz | docker load

```

## 10. 回滚

应用回滚：

1. 保留上一个可用镜像包。
2. 将旧镜像包放回 `images/info-ai-images.tar.gz`。
3. 执行 `./deploy.sh prod image-tar`。

数据库回滚：

1. 停止应用服务。
2. 确认备份文件。
3. 恢复备份。

```bash
cd /opt/info-ai
docker compose stop info-serve info-aggregation info-mvp info-admin

mysql -h 127.0.0.1 -uroot -p -e "DROP DATABASE IF EXISTS \`info-max\`; CREATE DATABASE \`info-max\` DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"
mysql -h 127.0.0.1 -uroot -p info-max < backup-info-max-YYYYMMDDHHMMSS.sql

docker compose up -d
```

注意：涉及数据库结构变更的回滚必须优先评估数据兼容性，不要在没有备份的情况下执行迁移。

## 11. 常见问题

### 容器启动但 API 连不上数据库

检查：

```bash
docker compose logs --tail=200 info-serve
docker compose logs --tail=200 info-aggregation
```

重点看：

- `.env` 中 `DB_PASSWORD` 和 `INFO_SERVE_MYSQL_DSN` 是否正确。
- MySQL 是否允许宿主机本地连接。
- `host.docker.internal` 是否能从容器解析。

### 管理后台能打开但接口失败

检查：

- `info-admin/nginx.conf` 的 `/api` 代理。
- `info-serve` 是否健康。
- 浏览器 Network 面板里 `/api/v1/*` 是否返回 200。

### 用户端事件为空

检查：

- 数据库是否已经执行 `mysql_migration_max.sql`。
- `crawl_task` 是否有 active 任务。
- `info_aggregation` 是否可以连上 MySQL 和 Redis。
- 是否需要手动触发核心渠道采集。

### 弱渠道可用率低

这是当前已知风险，不建议放宽首页质量门槛。优先处理：

1. 管理后台更新微博、小红书、知乎凭证。
2. 执行来源质量优先治理。
3. 分批处理 pending 队列。
4. 为社交热度召回更多二跳事实源。
