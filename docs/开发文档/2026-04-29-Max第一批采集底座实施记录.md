# Max 第一批采集底座实施记录

## 实施范围

本批次优先落地三类基础能力：

1. 管理后台可配置采集间隔。
2. 详情抓取升级的策略链骨架。
3. 详情低分补偿队列。

## 已完成改造

### 1. 渠道采集间隔管理化

- `channel` 表新增 Max 调度字段：
  - `base_interval_minutes`
  - `hot_interval_minutes`
  - `min_interval_minutes`
  - `max_interval_minutes`
  - `manual_interval_enabled`
  - `effective_interval_minutes`
  - `schedule_version`
- Python ORM `Channel.to_dict()` 已输出上述字段。
- Python API `/api/admin/channels` 已支持创建和更新上述字段。
- 调度同步 `_sync_crawl_tasks()` 改为使用实际生效间隔。
- 为历史渠道保留兼容逻辑：如果 Max 字段仍是默认值，但旧 `crawl_interval` 已配置，则继续使用旧间隔，避免升级后意外降频。
- Go 服务 `info-serve` 已透传 Max 调度字段，创建渠道时写入初始 `schedule_version=1`，更新渠道时自动 `schedule_version + 1`。
- 管理后台渠道页已增加基础间隔、热点间隔、生效间隔、最小/最大间隔和人工配置开关。
- 新增 MySQL 升级脚本：`docs/数据库/2026-04-29-Max渠道采集间隔配置迁移.sql`。

### 2. 详情策略链骨架

- 新增 `services.detail_strategy_chain`：
  - `DetailContext`：统一详情策略输入。
  - `DetailStrategy`：详情策略协议。
  - `DetailStrategyChain`：按低成本到高成本策略执行，拿到完整正文后短路。
- 当前骨架不引入 HTTPX、Trafilatura、Playwright 等依赖，只先固化接口和编排规则。
- 后续可按计划逐步接入：
  - API/移动端 JSON 策略
  - Trafilatura 正文抽取策略
  - CSS/XPath 渠道选择器策略
  - 搜索补全策略
  - Playwright/Crawl4AI 高价值兜底策略

### 3. 详情补偿队列

- 新增 `detail_job` 表和 ORM 模型，用于保存 `pending/list_only/failed/低分 partial` 内容的二次抓取任务。
- 新增 `services.detail_jobs.enqueue_low_quality_detail_jobs()`：
  - 扫描低分或失败详情内容。
  - 创建 `pending` 补偿任务。
  - 跳过已有 `pending/running` 任务，避免重复入队。
  - 根据低详情分设置较高优先级。
- 新增升级脚本：`docs/数据库/2026-04-29-Max详情补偿队列迁移.sql`。
- 新增 `services.detail_job_worker.process_pending_detail_jobs()`：
  - 按优先级和 `next_run_at` 取出 `pending` 任务。
  - 支持注入详情 runner，便于后续接入真实 crawler 和策略链。
  - 成功时更新 `Info.content/detail_fetch_status/detail_score/detail_strategy/detail_fetched_at`。
  - 失败时增加尝试次数，未达上限则延迟重试，达到上限后标记 `failed`。
  - 每次执行都会写入 `InfoAcquisitionLog`，保证补偿过程可追踪。
- 新增 `crawler_detail_runner()`，默认使用 `crawler_registry` 查找对应渠道爬虫并调用 `safe_fetch_detail()`。
- 当渠道爬虫返回失败、空正文或低分结果时，`crawler_detail_runner()` 会进入 HTTP HTML 兜底策略，使用 `HttpHtmlDetailStrategy` 拉取详情页并抽取正文。
- scheduler 新增 `process_detail_jobs()`：
  - 先调用 `enqueue_low_quality_detail_jobs()` 入队低分详情。
  - 再调用 `process_pending_detail_jobs()` 消费 pending 任务。
  - 默认每 10 分钟执行一次，作为 Max 详情补偿的第一条自动化闭环。

### 4. HTML 正文抽取器

- 新增 `services.html_article_extractor.HtmlArticleExtractor`。
- 当前实现不引入新依赖，优先抽取 `<article>`、常见正文容器和 `<main>`，最后回退 `<body>`。
- 该抽取器是 Trafilatura 接入前的稳定兜底；后续可以在同一接口下增加 Trafilatura 优先策略。
- 新增 `StaticHtmlArticleStrategy`，用于样本回放和策略链测试，真实 HTTP 策略后续可复用同一返回结构。
- 新增 `HttpHtmlDetailStrategy`：
  - 默认使用 HTTPX 获取 HTML。
  - 使用 `HtmlArticleExtractor` 抽取正文。
  - 支持测试注入 fetcher，避免单测依赖真实网络。
  - 当前作为详情补偿 worker 的低成本兜底策略。

### 5. 详情样本回放

- 新增 `services.detail_replay`，支持从 manifest 加载 HTML/JSON 样本并复用正式 `detail_pipeline` 做质量判断。
- 新增三类核心渠道样本：
  - 微博 JSON：覆盖热搜话题状态、长文和开放计划等关键字段。
  - 36 氪 HTML：覆盖文章正文容器和登录壳页噪声过滤。
  - CSDN HTML：覆盖 `main/article` 正文和推荐/登录噪声过滤。
- 样本回放测试要求命中关键正文词，并排除典型壳页噪声，后续策略改动必须保持这些样本可读。

### 6. 详情补偿队列报告

- 新增 `services.detail_job_report.build_detail_job_report()`。
- 新增 `/api/admin/detail-jobs` 管理接口，返回：
  - 队列总量。
  - 按状态统计。
  - 按渠道统计。
  - Top 失败原因。
  - pending 样本。
  - failed 样本。
- 该接口用于后续管理后台展示补偿积压、失败原因和渠道问题优先级。
- `info-serve` 已透传 `/api/v1/admin/detail-jobs`，并提供 `DetailJobReport`、`DetailJobSample` 等类型。
- `info-admin` 已新增“详情补偿”页面，展示队列总量、待处理/失败状态、渠道分布、Top 失败原因和 pending/failed 样本。
- `info-serve` 新增单条详情补偿任务操作接口：
  - `POST /api/v1/admin/detail-jobs/{id}/retry`：将 pending/running/failed 任务重新入队，立即进入下一轮补偿。
  - `POST /api/v1/admin/detail-jobs/{id}/cancel`：将 pending/running/failed 任务标记为 cancelled，用于人工止损。
- `info-admin` 详情补偿页面已接入人工操作：
  - pending 样本支持取消。
  - failed 样本支持重试和取消。
  - 操作后自动刷新队列报告，方便观察积压变化。
- 详情补偿报告已支持按 `channel_code` 和 `failure_reason` 过滤，便于聚焦单个渠道或单类失败。
- `info-serve` 新增 `POST /api/v1/admin/detail-jobs/retry` 批量重试接口，要求至少传入渠道或失败原因过滤条件，避免误操作全量队列。
- `info-admin` 详情补偿页面已增加筛选区：
  - 按渠道筛选。
  - 按失败原因筛选。
  - 基于当前筛选条件批量重试，单次最多 50 条。
- `info-serve` 新增 `GET /api/v1/admin/detail-jobs/{id}`，用于查看补偿任务详情、原文链接、当前正文和失败上下文。
- `info-serve` 新增 `POST /api/v1/admin/detail-jobs/cancel`，支持基于渠道或失败原因批量取消任务。
- `info-admin` 详情补偿页面已增加任务详情查看和批量取消。

### 7. 调度热更新

- `crawl_task` 新增 `schedule_version` 字段，记录已经同步的渠道调度配置版本。
- `_sync_crawl_tasks()` 同步渠道时会写入当前 `channel.schedule_version`。
- 当渠道 `effective_interval_minutes` 或 `schedule_version` 变化时，调度器会刷新：
  - `crawl_task.schedule_value`
  - `crawl_task.schedule_version`
  - `crawl_task.next_run_at`
- 新增升级脚本：`docs/数据库/2026-04-29-Max采集任务调度版本迁移.sql`。

### 8. Trafilatura 详情抽取策略

- `HttpHtmlDetailStrategy` 默认改为优先使用 `TrafilaturaArticleExtractor`。
- `TrafilaturaArticleExtractor` 策略顺序：
  - 优先调用 Trafilatura 抽取高质量正文。
  - 如果依赖不可用、抽取异常或结果过短，则自动回退到本地 `HtmlArticleExtractor`。
- `requirements.txt` 已新增 `trafilatura>=1.12.0`。
- 当前开发环境已安装并验证 Trafilatura `2.0.0`。
- 样本回放和详情补偿 worker 继续复用同一策略链，避免线上策略和测试策略割裂。

### 9. 新鲜度监控入口

- 管理首页已接入 `channel-health` 数据并展示“新鲜度风险”。
- 新鲜度风险规则：渠道最近 6 小时没有 `latest_info_at` 更新，或时间无法解析，则进入风险列表。
- 首页新增：
  - 新鲜度风险数量指标。
  - 风险渠道列表，展示最新入库时间、健康等级、健康分和最近问题。
- 该入口复用已有渠道健康数据，不新增表结构，便于快速验收和后续扩展 SLA 配置。

## 后续建议

1. 将管理后台渠道编辑能力从“新增”扩展为“新增 + 编辑”，让现有渠道也能在线调整 Max 调度字段。
2. 为 `schedule_version` 增加调度器热更新检测，而不只是写入版本号。
3. 接入 HTTPX + Trafilatura 的第一条真实详情策略，并用微博、36 氪、CSDN 样本做回放测试。
4. 持续补充真实线上样本，将每个核心渠道扩展到至少 20 条 HTML/JSON 回放样本。
5. 为“详情补偿”页面继续增加批量取消、任务详情弹窗和失败原因趋势图。
