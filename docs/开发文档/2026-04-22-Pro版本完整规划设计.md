# Pro 版本完整规划设计

更新时间：2026-04-22

## 版本定位

Pro 版本是“信息达人”从 MVP/Plus 阶段走向可持续产品化架构的关键版本。

Plus 版本已经完成了移动端体验打磨、数据质量治理、体育数据源补充、首页筛选记忆等能力。Pro 版本不再只是继续加功能，而是要完成一次系统架构升级：

1. 从单体后端演进为采集服务、业务服务、用户前端、管理后台分层架构。
2. 从 SQLite 迁移到 MySQL，建立可长期维护的数据底座。
3. 引入用户鉴权系统，让普通用户和管理员拥有不同能力边界。
4. 建立独立 PC 管理与监控后台，把配置管理、采集监控、数据质量治理从用户端拆出去。
5. 建设 Go 业务服务 `info-serve`，作为统一对外 API 层。

## 产品架构

Pro 版本建议形成四个核心模块：

```text
info_aggregation
  数据采集服务
  负责爬虫、清洗、详情补全、质量评估、事件构建、写入 MySQL

info-serve
  Go 业务服务
  负责对外 API、用户鉴权、业务查询、权限控制、用户偏好、收藏、订阅

info-max
  用户端前端
  移动端优先，普通用户不强制登录，登录后解锁更多个性化能力

info-admin
  管理监控后台
  PC Web 优先，必须登录，面向管理员和运营人员
```

## 服务边界

### info_aggregation

定位：数据生产者。

核心职责：

- 渠道爬虫。
- 数据清洗。
- 详情页补全。
- 数据质量评分。
- 低质量内容过滤。
- 事件构建。
- 采集日志记录。
- 定时任务调度。
- 写入 MySQL。

不再承担：

- 用户端公开 API。
- 用户鉴权。
- 用户收藏、订阅、偏好。
- 管理后台页面能力。

### info-serve

定位：业务服务和对外 API。

核心职责：

- 提供用户端 API。
- 提供管理后台 API。
- 用户注册、登录、退出。
- 管理员登录和权限校验。
- 查询事件列表、详情、时间线、来源。
- 查询分类、渠道、数据质量摘要。
- 用户收藏、关注主题、筛选偏好。
- API 返回结构统一治理。
- 后续支持缓存、限流、审计日志。

### info-max

定位：用户端产品。

核心职责：

- 面向普通用户展示热点事件。
- 移动端优先。
- 不强制登录。
- 未登录用户可浏览热点、搜索、筛选、查看详情。
- 登录用户可使用收藏、关注、订阅、个性化排序、跨设备偏好同步等能力。

### info-admin

定位：管理与监控后台。

核心职责：

- PC Web 优先。
- 必须登录。
- 仅管理员和运营人员可访问。
- 管理分类、渠道、爬取频率、启停状态。
- 手动触发爬取。
- 查看采集日志、失败原因、完整率趋势。
- 查看数据质量报告。
- 查看低质量数据、重复数据、详情不完整数据。
- 触发数据归档、事件重建、单条重采。

## 用户鉴权设计

### 用户注册方式

Pro 第一阶段仅支持邮箱注册。

注册流程：

1. 用户输入邮箱和密码。
2. 系统校验邮箱格式和密码强度。
3. 创建用户。
4. 发送邮箱验证码或验证链接。
5. 用户完成邮箱验证。
6. 登录后创建会话。

第一阶段可以先使用邮箱验证码模拟或本地开发模式，后续再接入真实邮件服务。

### 用户登录能力

未登录用户可以：

- 查看首页事件。
- 切换分类。
- 搜索事件。
- 查看时间线。
- 查看详情。

登录用户额外可以：

- 收藏事件。
- 跨设备保存筛选偏好。
- 关注主题或关键词。
- 查看阅读历史。
- 后续接收订阅提醒。

### 管理后台鉴权

管理后台必须登录。

访问规则：

- 未登录访问 `info-admin` 自动跳转登录页。
- 普通用户登录后不能访问管理后台。
- 管理员可以访问渠道管理、分类管理、采集任务、质量报告。
- 所有管理操作写入 `admin_audit_log`。

角色建议：

- `user`：普通用户。
- `admin`：管理员。
- `operator`：运营人员，可管理采集和数据质量，但不能管理用户权限。

## info-serve 技术架构

建议语言：Go。

建议目录：

```text
info-serve/
  cmd/server/
    main.go
  internal/
    config/
    database/
    router/
    middleware/
    handler/
    service/
    repository/
    model/
    dto/
    auth/
    errors/
    logger/
  migrations/
  docs/
  go.mod
  README.md
```

推荐技术选型：

- Web 框架：`gin`。
- MySQL 驱动：`go-sql-driver/mysql`。
- SQL 访问：第一阶段推荐 `sqlx` 或标准库 SQL，避免过早引入重 ORM。
- 配置：`.env` + 配置结构体。
- 鉴权：服务端 session 或 JWT。第一阶段推荐服务端 session，更方便管理后台注销和权限回收。
- 密码：`bcrypt` 哈希存储。
- 日志：结构化日志。
- API：RESTful，统一 `/api/v1` 前缀。

## 风险与对策

### 数据迁移风险

风险：SQLite 和 MySQL 字段类型、时间格式、唯一约束行为不完全一致。

对策：

- 先生成 schema。
- 小批量迁移。
- 校验数量。
- 校验重复。
- 校验事件重建结果。
- 保留 SQLite 备份。

### 服务拆分风险

风险：前端切换 API 后出现行为差异。

对策：

- `info-serve` 返回结构尽量兼容现有前端。
- 先并行运行 Python API 和 Go API。
- 前端通过环境变量切换 API。
- 完成回归测试后再切默认入口。

### 鉴权复杂度风险

风险：用户系统会引入安全和状态管理复杂度。

对策：

- 第一阶段只支持邮箱注册。
- 管理员账号先通过数据库初始化或脚本创建。
- 密码必须哈希。
- session 必须可失效。
- 管理接口必须全量鉴权。

### 管理后台范围膨胀

风险：管理后台容易无限扩展。

对策：

- 第一阶段只做采集和质量闭环。
- 用户管理、权限细分、复杂报表放到后续。

## 推荐优先级

我建议 Pro 按以下优先级推进：

1. MySQL 建表和迁移。
2. `info-serve` Go 服务只读 API。
3. `info-max` 切换 API。
4. 用户鉴权和服务端偏好。
5. 独立 `info-admin` 管理后台。
6. 管理后台采集监控和数据质量治理。

原因：

- 数据库是所有服务拆分的基础。
- Go 服务先做只读 API，风险最低。
- 前端先平滑切换，再做登录增强。
- 管理后台最后做，可以复用前面已经稳定的鉴权和 API 基础。
