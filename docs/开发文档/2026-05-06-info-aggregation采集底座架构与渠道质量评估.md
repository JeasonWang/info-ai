# info-aggregation 采集底座架构与渠道质量评估

日期：2026-05-06

## 1. 总体结论

`info-aggregation` 当前已经不是简单的“抓标题列表”服务，它具备了采集底座的核心雏形：渠道爬虫抽象、定时任务、管理端可配置采集间隔、详情二次抓取、详情质量评分、低质量补偿队列、采集日志、数据质量快照、事件重建等能力。

从架构成熟度看，当前状态可以评价为“中上，可继续演进为高质量数据底座”。它已经开始按渠道做差异化采集，但还没有达到“所有渠道都稳定抓到完整、有价值、可分析正文”的标准。主要短板集中在登录态治理、反爬适配、浏览器资源池、渠道级质量验收、真实线上样本回放、弱渠道的全文来源能力上。

当前热点数据采集方向是正确的：先抓热点入口，再补详情正文，再做质量评分和补偿。但热点入口和详情完整度不能混为一谈。微博、知乎、小红书这类社交平台，热点入口通常只给短文本或话题线索，必须二次搜索、详情页渲染、长文接口、登录 Cookie 才能拿到可分析内容；路透这类国际媒体当前更偏“官方线索恢复”，不能把 sitemap 或新闻索引摘要误判为完整正文。

## 2. 当前架构优点

1. `BaseCrawler` 统一了 HTTP Session、请求头、重试、JSON 获取、详情安全兜底等基础能力，降低了每个渠道重复实现成本。
2. `CrawlerRegistry` 提供渠道注册和渠道级锁，避免同一渠道被手动采集、定时任务、补偿任务同时打爆。
3. 调度器已经从“固定分类间隔”演进到按 `CrawlTask.next_run_at` 判断 due 渠道，管理页配置的单渠道采集间隔可以真正影响采集节奏。
4. 保存链路会记录 `detail_fetch_status/detail_score/detail_strategy/detail_content_length/detail_fetch_error`，方便后续诊断“为什么详情低分”。
5. `detail_pipeline` 已经引入渠道 Profile，不同渠道有不同的完整度阈值、最小段落数、价值关键词和壳页面识别规则。
6. 已经有 `DetailJob` 补偿队列和 `DataQualitySnapshot` 质量快照，说明底座开始具备运营治理能力。
7. 多个渠道已经不是单次抓取，而是有 API、网页、内嵌状态、搜索、浏览器渲染等多策略链路。

## 3. 当前架构主要问题

1. 爬虫仍以全局实例注册，实例内部持有 `requests.Session`。渠道锁降低了同渠道并发风险，但从长期看，Session、Cookie、Headers、失败状态最好限定在单次采集上下文，避免未来多进程、多 worker、补偿任务并行时出现状态污染。
2. Cookie、签名、登录态读取散落在微博、知乎、小红书等爬虫内部。后续应该抽象 `CredentialProvider`，统一支持 Cookie 有效性检测、过期告警、脱敏日志、渠道凭证优先级。
3. Playwright 是按渠道临时启动，缺少浏览器池和上下文复用。今日头条、知乎、小红书一旦批量触发渲染，采集速度、资源占用、被风控概率都会明显上升。
4. 反爬失败原因没有形成统一枚举。例如 401、403、滑块验证、登录页、空壳 HTML、地区限制、新闻索引跳转都应该有标准错误码，便于管理端判断“是站点变了，还是 Cookie 失效，还是需要代理”。
5. 真实线上样本回放不足。采集服务的验收不应该只看接口返回成功，而要固定每个渠道 20 条真实样本，持续记录正文长度、完整率、低分原因、失败策略。
6. 事件重建仍依赖 `core_entity/title` 的浅层聚合，内容越短越容易把不同事件聚错；底座应该优先保证详情可读，再考虑事件智能化。
7. 评分仍偏启发式。长度、关键词、段落数能筛掉明显垃圾，但无法完全判断“是否有分析价值”。后续需要加入来源权威度、热度、时效、实体密度、事件信息量等指标。

## 4. 热点数据方式是否正确

当前方式总体正确，但需要分渠道定义“热点合同”：

1. 热点入口必须是真正的榜单、热搜、首页重点、推荐热榜或官方新闻最新入口，不能只抓随机首页链接。
2. 列表数据只作为线索，不作为最终可分析正文。
3. 每个渠道必须有详情策略链：轻量 API 或内嵌 JSON 优先，网页正文其次，搜索补充和浏览器渲染作为兜底。
4. 对需要登录的平台，要明确“无 Cookie 可采集到什么、有 Cookie 可提升到什么、Cookie 失效如何报警”。
5. 对天然短内容渠道，不能简单用文章类正文长度标准，而要补充热度、评论、官方回应、代表性帖子、相关内容聚合。

## 6. 渠道特殊处理现状

当前已经做了不少渠道特殊处理：

1. 微博：接入 `WEIBO_COOKIE`，尝试移动端接口、长微博、转发原文、话题搜索。
2. 今日头条：使用热榜 API，并引入搜索补充和 Playwright 渲染兜底。
3. 知乎：接入 `ZHIHU_COOKIE/ZHIHU_ZSE_93/ZHIHU_ZSE_96`，通过搜索接口找到真实问题或文章，再抓回答详情。
4. 小红书：接入 `XHS_COOKIE`，解析 `INITIAL_STATE`，补充标签和互动指标，必要时浏览器渲染。
5. 36氪：使用热榜 API、网页 initialState、详情 API，遇到挑战页时用新闻索引恢复线索。
6. Reuters：使用 sitemap 和 Google News 兜底，避免在官方入口不可访问时伪造正文。
7. 技术文章类：CSDN、掘金、博客园都针对正文容器、API 内容、内联状态做了独立解析。
8. 体育类：央视体育和新浪体育都做了 URL 类型过滤、标题过滤、编码修正、正文块提取。

这些特殊处理说明方向是对的，但还需要把“特殊处理”产品化为可观测、可验收、可维护的策略链，而不是散落在各个爬虫文件里。

## 7. 优化方向

### P0：建立真实采集验收体系

1. 每个渠道固定采集 20 条真实线上数据，保存样本标题、来源 URL、详情策略、详情长度、详情分数、失败原因。
2. 管理端增加渠道质量看板：完整率、低分率、平均详情长度、失败策略 TOP、Cookie 健康状态。
3. 明确 seed/mock 数据不得进入真实采集质量统计。
4. 为每个渠道建立最低验收线：文章类渠道完整率不低于 70%，社交热点渠道完整或高分 partial 不低于 60%，弱源渠道必须明确标记为线索级。

### P1：重构采集策略链

1. 抽象 `DetailStrategyChain`，每个策略输出统一字段：strategy、status、score、content_length、failure_code、latency_ms。
2. 轻量策略达到阈值后立即短路，浏览器渲染只作为最后兜底。
3. 建立浏览器池，复用 browser/context，支持渠道级并发上限。
4. 建立 `CredentialProvider`，统一读取 `.env`、本地环境变量、容器环境变量，统一脱敏和过期检测。
5. 建立反爬响应分类：login_required、captcha_required、forbidden、empty_shell、rate_limited、geo_blocked、api_changed。

### P2：重点补弱渠道

1. Reuters：优先解决全文源。可选方案包括授权 feed、稳定代理、Reuters article API 参数修复、或者引入 AP/BBC/CNN/Guardian 等替代国际源。当前 sitemap 只能作为线索，不应算完整正文。
2. 微博：source_id 改为规范化话题 URL、mid 或搜索结果真实对象 ID；代表性内容优先官方媒体、蓝 V、互动高、时间新的微博。
3. 小红书：入口从 explore 扩展为关键词/topic 搜索；过滤低字数纯图片笔记；把互动数、标签、发布时间纳入价值评分。
4. 知乎：增加 Cookie/ZSE 健康检测；区分 question、answer、article、zvideo；答案排序综合赞同数、正文长度、发布时间。
5. 今日头条：区分热榜事件和文章页；搜索补充时优先媒体文章、发布时间新、标题相关度高的结果。
6. 36氪：持续跟踪详情 API 返回结构；遇到挑战页时明确标记，不把索引摘要当完整正文。

### P3：提高价值评分

1. 价值评分从“长度 + 关键词”升级为“来源权威度 + 热度 + 新鲜度 + 正文信息密度 + 实体数量 + 事件增量”。
2. 社交渠道增加代表性样本聚合，而不是只保存单条短正文。
3. 财经和体育类增加结构化字段：指标变化、赛事、比分、球队、时间、人物。
4. 技术渠道增加技术实体识别：语言、框架、模型、公司、产品、版本号。

### P4：工程治理

1. 单次采集上下文化，避免全局爬虫实例长期持有可变 Session。
2. 增加渠道级限流、熔断、指数退避、失败冷却。
3. 将每个渠道的入口 URL、请求头、是否需要 Cookie、完整度阈值、启用策略写入配置表或配置文件。
4. 增加真实样本回放测试，避免站点结构变化后静默降级。
5. 增加采集日志关联 trace_id，让一次列表抓取、详情抓取、补偿任务、入库记录能串起来。

## 8. 下一步建议

下一阶段不建议盲目增加更多渠道，而应先把现有渠道做成可验收的高质量样板。建议优先顺序：

1. 先做“渠道质量看板 + 真实样本验收脚本”，让问题可见。
2. 再做 `CredentialProvider + DetailStrategyChain`，让登录态和详情链路可维护。
3. 接着重点优化 Reuters、微博、小红书、知乎这四个最容易低分的渠道。
4. 最后把价值评分和事件重建升级，确保前端看到的是完整、有分析价值、可追踪来源的数据。

判断 `info-aggregation` 是否达到 Max 版本采集底座标准，不应该只看“每个渠道能返回 20 条”，而应该看“每个渠道返回的 20 条中，有多少条是真实、最新、热点、正文完整、可解释、可分析、可追溯的内容”。这是后续所有优化的核心验收标准。
