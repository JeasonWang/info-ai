# Max 事件分析能力架构升级计划

## 背景

Max 版本已经上线，信息达人后续不再维护 `info-max`，用户端统一走 `info-mvp`。当前产品的第一竞争力是采集完整、有价值的数据，第二竞争力是把热点数据加工成清晰可信的事件分析。

线上观察到的问题是：`event.one_line_summary`、`event_summary_snapshot.content`、`event_timeline_entry.summary` 等字段仍存在简单裁剪正文的痕迹，容易出现断句、语义不完整、作者和阅读时间等网页元信息混入摘要，以及多个详情模块内容重复。这个问题不能只在前端隐藏，需要在事件分析底座中解决。

## 升级目标

1. 将 `info_aggregation` 内部拆成两条明确链路：
   - 数据采集链路：负责抓取、详情补偿、质量评分、原文保存。
   - 事件分析链路：负责事实抽取、事件摘要、时间线、价值判断、风险提示和模型增强。
2. 事件分析必须有两套能力：
   - 规则兜底：不依赖大模型，保证任何环境都能生成通顺、不截断、可解释的分析。
   - 大模型增强：后续接入本地千问或商用模型，对规则结果进行增强、改写和补充。
3. 数据库一次性预留事件分析所需表结构，后续迭代不再频繁破坏表结构。
4. 事件分析结果必须可追踪：知道用了哪些来源、哪个引擎、置信度如何、有哪些质量风险。

## 架构设计

### 1. 数据采集线

数据采集线继续维护现有职责：

- `Info` 保存原始热点和详情正文。
- `InfoAcquisitionLog` 记录详情采集路径和质量。
- `DetailJob` 负责低质量详情补偿。
- `acquisition_quality` 负责完整度、价值性、风险原因判断。

这条线只负责把内容抓完整、抓真实，不负责生成面向用户的事件解读。

### 2. 事件分析线

新增 `services/event_analysis/` 包，包含：

- `schemas.py`：事件分析输入、事实包、分析结果、时间线点等结构。
- `text_utils.py`：正文清洗、句子切分、自然断句、重复判断。
- `fact_extractor.py`：从多个来源中抽取核心事实、关键词、时间线候选和证据。
- `rule_provider.py`：规则兜底分析器，生成稳定的摘要、原因、风险和来源对比。
- `providers.py`：大模型 Provider 抽象和多模型适配器。
- `validator.py`：校验分析结果，防止空摘要、截断、标题复读、字段重复。
- `pipeline.py`：统一编排规则分析、大模型增强、校验和回退。

事件构建器 `event_builder.py` 不再直接裁剪正文生成分析，而是调用事件分析管线。

## 大模型适配策略

第一阶段不强依赖模型，先把接口和回退机制做好。后续本地部署千问时，优先使用 OpenAI Compatible 协议，因为 vLLM、Xinference、LM Studio、Ollama OpenAI 兼容接口都能复用。

预留配置：

```env
EVENT_ANALYSIS_MODE=hybrid
EVENT_ANALYSIS_ENABLE_LLM=0
EVENT_ANALYSIS_PROVIDER=openai_compatible
EVENT_ANALYSIS_BASE_URL=http://127.0.0.1:8001/v1
EVENT_ANALYSIS_API_KEY=
EVENT_ANALYSIS_MODEL=qwen2.5-14b-instruct
EVENT_ANALYSIS_TIMEOUT=60
EVENT_ANALYSIS_MAX_INPUT_CHARS=12000
EVENT_ANALYSIS_TEMPERATURE=0.2
EVENT_ANALYSIS_FALLBACK_TO_RULE=1
```

支持 Provider：

- `rule`：纯规则，默认兜底。
- `openai_compatible`：本地千问、vLLM、Xinference、LM Studio、Ollama OpenAI 兼容接口。
- `ollama`：预留 Ollama 原生 `/api/chat` 接口。
- `dashscope`：预留通义千问云端接口。
- `custom_http`：预留自定义 HTTP 网关。

## 分阶段实施

### 第一阶段：规则分析底座

1. 新增事件分析数据结构和文本工具。
2. 用测试锁定：摘要不能简单截断，不能包含作者/阅读时长元信息，时间线摘要必须自然断句。
3. 实现规则事实抽取和规则分析器。
4. 将 `event_builder.py` 接入分析管线。
5. 写入新分析表，同时同步旧表。

### 第二阶段：大模型接口预留

1. 新增 Provider 抽象。
2. 实现 OpenAI Compatible Provider。
3. 实现 Prompt 和 JSON 输出解析。
4. 校验失败时自动回退规则结果。
5. 默认关闭 LLM，配置开启后可直接请求 `http://127.0.0.1:8001/v1`。

### 第三阶段：分析质量治理

1. 为每次分析生成质量分和风险原因。
2. 管理端展示低质量事件分析。
3. 针对低质量事件支持重新分析。
4. 对采集不完整导致的分析低分，反向创建详情补偿任务。

### 第四阶段：产品展示增强

1. `info-mvp` 详情页按分析类型展示：发生了什么、为什么重要、最新进展、来源对比、风险提示、证据链。
2. 隐藏重复模块，不展示无信息量字段。
3. 支持“AI 增强版”和“规则兜底版”的内部质量追踪。

## 验收标准

1. 单来源长文事件：一句话摘要不能是正文前 120 字裁剪，不能以半句话结尾。
2. 多来源事件：摘要能体现跨来源跟进，不重复标题。
3. 时间线：每个节点是完整句子，不混入作者、阅读时间、浏览量等网页元信息。
4. 数据库：新分析表存在，旧表仍有兼容数据。
5. LLM 未开启时：事件分析能稳定运行。
6. LLM 开启但失败时：自动回退规则结果，不影响事件重建。

## 本轮开发范围

本轮直接完成第一阶段和第二阶段的底座代码：数据库表、ORM 模型、规则分析管线、Provider 抽象、OpenAI Compatible 适配、事件构建接入和测试。管理端展示和用户端进一步增强放到下一轮。

### 当前闭环

```text
事件分析质量低
  -> 事件分析质量页点击“入队弱来源补偿”
  -> 创建 detail_job
  -> 详情补偿成功，Info 正文更新
  -> 关联事件分析标记 stale
  -> 调度器自动重建事件分析，或管理端手动点击“处理过期分析”
  -> 新摘要、新时间线、新分析快照写入数据库
```

## v1.0.0 核心收敛与熔断实施记录

### v1.0.0 核心目标

当前版本先保证事件分析能力能稳定运行，非核心的运营统计、模型效果评测、多模型投票和复杂 prompt 工作台放到 v2.0.0。v1.0.0 必须闭环：

1. 管理端可以配置多个大模型。
2. 事件分析能真实调用启用的大模型。
3. 大模型返回异常、超时或输出不合格时，不影响事件重建。
4. 模型连续失败后自动熔断，后续分析跳过该模型。
5. 所有模型停用、超限或熔断时，自动使用本地规则分析。

### 已完成

1. `llm_model_config` 增加熔断字段：
   - `consecutive_failure_count`：连续失败次数。
   - `circuit_open_until`：熔断结束时间。
   - `last_failure_reason`：最近失败原因。
2. 新增 `llm_call_log` 表，记录每次模型调用：
   - 模型配置 ID、供应商、模型名。
   - 调用状态、耗时、输入来源数量、错误信息。
3. 事件分析管线已记录模型成功和失败：
   - 成功：递增当日调用次数，清空连续失败和熔断状态。
   - 失败：记录失败日志，递增调用次数和连续失败次数。
   - 连续失败达到 `EVENT_ANALYSIS_LLM_FAILURE_THRESHOLD` 后，设置 `circuit_open_until`。
4. 模型选择逻辑已支持熔断：
   - 跳过停用模型。

   - 近 100 次成功/失败次数。
