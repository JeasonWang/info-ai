# Max 数据采集与事件分析持续优化执行计划

## 目标

信息达人的核心竞争力是“完整、有价值、可分析”的热点事件数据。后续优化不再以单个爬虫临时修补为主，而是建设一套质量驱动的数据采集和事件分析底座，让系统能持续识别低质量数据、自动补偿详情、生成更有解释力的事件分析。

## 一、采集质量底座

1. 建立统一采集质量画像，输出正文完整度、信息价值、时效性、可信来源、风险原因和补偿建议。
2. 按渠道定义完整度规则。文章型渠道要求完整正文，社交型渠道要求话题背景和互动语境，问答型渠道要求问题与代表回答，财经体育新闻型渠道要求事实、指标或结果。
3. 所有质量判断统一从质量画像读取，避免 `detail_jobs`、质量报告、事件构建各自使用不同阈值。

## 二、详情补偿闭环

1. 对 `failed`、`list_only`、`partial`、短正文、低评分、反爬壳页面自动入队。
2. 补偿任务按失败原因和渠道类型排序：反爬登录态优先提示配置 Cookie，短正文优先走详情页/API/渲染兜底。
3. 每次补偿后更新质量画像，让管理端能看到补偿是否真正提升完整度。

## 三、事件分析升级

1. 事件摘要从“拼接短文本”升级为结构化分析快照。
2. 每个事件至少生成：一句话看懂、发生了什么、为什么重要、最新进展、为什么热、风险提示、来源对比。
3. 分析内容必须携带质量意识：来源少、详情不完整、疑似列表摘要时，要明确提示风险，不能包装成确定结论。

## 四、产品展示升级

1. `info-mvp` 事件详情页继续区分摘要和代表原文，后续增加“为什么热”“风险提示”“多平台观点”。
2. 管理后台质量报告展示每个渠道可用率、完整率、平均正文长度、主要失败原因、弱样本。
3. 运营动作聚焦“重抓低完整详情”“补 Cookie”“重建事件”“归档低质量内容”。

## 五、实施顺序

1. 第一批：统一质量画像、补偿队列接入、质量报告接入、事件分析新增热度原因和风险提示。
2. 第二批：按渠道重构详情抓取策略，逐个渠道做线上样本对比验收。
3. 第三批：升级事件聚合算法，引入实体、时间、语义和跨渠道证据。
4. 第四批：前端热点分析页升级，展示结构化事件分析结果。

## 当前批次验收标准

1. 同一条数据在补偿队列和质量报告中的完整度判断一致。
2. 文章型渠道短正文即使评分中等，也会进入补偿。
3. 事件详情 API 能返回“为什么热”和“风险提示”快照。
4. 质量报告弱样本能说明具体失败原因和建议动作。
