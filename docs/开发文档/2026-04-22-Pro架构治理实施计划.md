# Pro 架构治理实施计划

更新时间：2026-04-22

## 目标

本计划用于修正 Pro 阶段的工程架构问题：将 `info-admin` 从静态原型升级为标准 Vue3 + TypeScript 管理后台，同时规范 `info-serve` Go 服务架构，避免后续服务继续以临时实现方式演进。

## 实施原则

- 先架构设计，后代码实现。
- 每个服务必须能独立运行、测试、构建、部署。
- 新增代码文件统一英文命名。
- 页面展示和文档使用中文。
- 业务代码注释使用中文解释关键业务意图。
- 每个阶段必须有可验证结果。
- 每个阶段独立提交，避免大爆炸式改动。

## 阶段三：info-serve HTTP 层迁移准备

状态：已完成。

### 目标

为后续 `internal/transport/http` 迁移做准备，但不在本阶段大规模搬文件，避免风险过大。

### 任务

1. 在架构文档中标注现有 `handler/router/middleware` 为待迁移目录。
2. 新增迁移说明，约定后续新 HTTP 代码进入 `internal/transport/http`。
3. 新增代码审查规则：新增 handler 不允许直接写 SQL。

完成记录：

- 新增 `internal/transport/http` 路由入口。
- `internal/app` 已直接依赖 `internal/transport/http`。
- `internal/router` 已收敛为历史兼容包装。
- 新增 transport 路由测试，覆盖健康检查和注册接口。
- `health` 和 `auth` handler 已迁入 `internal/transport/http`，旧 handler 文件已删除。
- `events` handler 已迁入 `internal/transport/http`，用户侧事件 API 入口已归位。
- 管理后台 handler 已迁入 `internal/transport/http`，`internal/handler` 已不再承载业务文件。
- 管理鉴权中间件已迁入 `internal/transport/http/middleware`，`internal/middleware` 已不再承载业务文件。

## 优先级建议

推荐先执行阶段一，再执行阶段二。

原因：

- `info-admin` 当前问题最明显，静态原型继续存在会误导后续开发。
- `info-admin` 工程化之后，可以直接消费 `info-serve` 已有接口。
- `info-serve` 当前结构虽然需要治理，但已有测试保护，可以在后台工程稳定后逐步收敛。

## 风险控制

- 不一次性重构 `info-admin` 和 `info-serve` 所有代码。
- 不改变 `info-serve` 已有 API 路径和响应结构。
- `info-admin` 重建期间不影响 `info-max` 用户端。
- 每个阶段验证通过后再提交。
