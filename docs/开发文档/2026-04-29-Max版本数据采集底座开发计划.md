# 信息达人 Max 版本数据采集底座开发计划

## 1. 版本目标

Max 版本的核心目标是把信息达人从“能聚合热点事件”升级为“稳定采集最新、最热、最完整、最有价值的数据底座”。当前 Pro 版本已经完成服务拆分、MySQL 迁移、管理后台、详情评分、事件流 MVP 和基础监控，但从代码和现有链路看，采集仍存在四类关键问题：

1. 数据内容不完整：详情页失败后多停留在列表摘要，失败项没有持续补偿。
2. 数据详情评分低：详情策略以渠道内解析为主，通用兜底和质量评估仍偏轻。
3. 数据采集不及时：调度主要按分类固定间隔串行执行，没有按热度、失败率、渠道时效动态调整。
4. 数据内容不够新：入库、事件重建、质量快照之间缺少以新鲜度为核心的 SLA 和告警闭环。

Max 版本要围绕“覆盖率、完整度、及时性、价值密度、可观测性”五个指标建设采集底座，让前端事件流和详情页有持续可用的高质量原始数据。

## 2. 现状判断

### 2.1 已具备的基础

- `info_aggregation` 已有统一 `BaseCrawler`、渠道 registry、清洗、入库、详情抓取和质量报告。
- 已支持微博、头条、小红书、东方财富、路透、CSDN、掘金、博客园、36 氪、知乎、央视体育、新浪体育等渠道。
- `detail_pipeline` 已能输出 `complete / partial / list_only / failed`、详情策略、详情分、正文长度和失败原因。
- `InfoAcquisitionLog`、`CrawlRunLog`、`DataQualitySnapshot` 已经具备基础采集监控能力。
- `event_builder` 已能从 `Info` 聚合事件，并按来源数、详情质量生成事件摘要和时间线。

### 2.2 主要短板

- 调度粒度偏粗：目前按分类固定间隔执行，分类内渠道串行跑，无法对热点渠道加速，也无法对失败渠道降频保护。
- 详情补偿不足：只对新入库数据即时抓详情，`pending/list_only/failed/低分 partial` 没有独立重试队列、优先级和退避策略。
- 渠道策略不分层：每个爬虫各自实现详情解析，缺少“官方 API、移动端 API、RSS/站点地图、HTML、搜索补全、浏览器渲染”等统一策略编排。
- 新鲜度无硬指标：数据有 `event_time/created_at/detail_fetched_at`，但没有 freshness SLA、延迟分布、超时告警和自动提频。
- 价值筛选偏弱：现在主要过滤低质量和重复内容，还没有结合热度、来源权威、跨源扩散、实体重要性、用户消费反馈做价值评分。
- 环境一致性需要补强：本地质量报告执行时发现默认 SQLite schema 缺少 `event.event_key`，说明 Max 版本必须把迁移校验和启动前健康检查作为底座能力。

## 3. Max 版本核心指标

| 指标 | 目标口径 | Max 验收目标 |
| --- | --- | --- |
| 渠道可用率 | 单渠道近 24 小时成功采集次数 / 应采集次数 | 核心渠道 >= 95% |
| 新鲜度 | 来源发布时间到入库时间 P90 | 热点 <= 10 分钟，其他 <= 30 分钟 |
| 详情完整率 | `complete` 且 `detail_score >= 70` 的内容占比 | 核心展示池 >= 80% |
| 低质内容率 | 空正文、标题即正文、反爬壳页、短正文占比 | <= 5% |
| 重复率 | 标题/URL/语义近重复内容占比 | <= 8% |
| 事件有效率 | 有多来源或高权威来源、摘要非空、时间线可读的事件占比 | >= 85% |
| 补偿恢复率 | 初次详情失败后 24 小时内补抓成功占比 | >= 60% |

## 4. 总体架构优化

Max 版本将采集底座拆成五层：

1. 渠道发现层：维护渠道能力档案，包括 API、网页、RSS、搜索页、是否需要渲染、限流策略、权威度、分类覆盖。
2. 抓取执行层：把列表抓取和详情抓取拆成可排队任务，支持优先级、并发、超时、退避、失败重试和按渠道限流。
3. 详情补全层：统一详情策略编排，按渠道先后尝试 API、移动端、结构化 JSON、HTML 主体抽取、搜索补全和可选浏览器渲染。
4. 质量评分层：在入库前后统一计算完整度、新鲜度、热度、价值、重复、可信度，并将低质内容隔离到观察池。
5. 监控运营层：把采集运行、详情失败、渠道健康、质量趋势、低分样本、补偿进度暴露给管理后台和告警任务。

建议新增或改造的核心模型：

- `channel_profile`：渠道能力、权威度、支持的策略、限流配置、健康状态。
- `crawl_job`：列表采集任务队列，支持优先级、状态、重试次数、下次执行时间。
- `detail_job`：详情补全任务队列，绑定 `info_id`，支持失败原因、策略链、退避时间。
- `content_quality_score`：内容质量评分明细，保存完整度、新鲜度、热度、价值、可信度、重复风险。
- `source_signal`：来源侧热度信号，例如热搜排名、阅读数、评论数、转发数、榜单位置、发布时间。

`channel_profile` 必须支持运营侧配置采集间隔，至少包含：

- `base_interval_minutes`：基础采集间隔，由管理后台手动配置。
- `hot_interval_minutes`：热点加速间隔，用于热搜、榜单、突发类渠道。
- `min_interval_minutes`：系统允许的最小间隔，防止误配置导致反爬或资源打满。
- `max_interval_minutes`：失败退避后的最大间隔。
- `manual_interval_enabled`：是否启用人工配置覆盖默认策略。
- `effective_interval_minutes`：当前实际生效间隔，由人工配置、热点加速、失败退避共同计算。
- `schedule_version`：配置版本号，管理后台修改后调度器按版本热更新。

## 5. 关键建设方向

### 5.1 完善渠道覆盖

第一批重点渠道按“热点、财经、科技 AI、体育、国际”拆分：

- 热点：微博热搜、头条热榜、百度热搜、抖音热点、知乎热榜、小红书热点。
- 财经：东方财富、财联社、华尔街见闻、证券时报、央行/统计局公开数据、重要商品行情。
- 科技 AI：36 氪、机器之心、量子位、InfoQ、CSDN、掘金、博客园、GitHub Trending、Hacker News。
- 体育：央视体育、新浪体育、腾讯体育、虎扑、懂球帝、NBA/英超/中超公开赛程与新闻。
- 国际：Reuters、AP、BBC、CNN、联合国/白宫/欧盟等官方新闻源。

每个渠道必须建立“列表源 + 详情源 + 备用源 + 质量指标 + 降级策略”。新增渠道不直接裸写爬虫，而是先写渠道画像，再选择通用策略组合。

### 5.2 详情抓取升级

详情链路从“爬虫内自行 resolve_detail”升级为统一详情管道：

1. 列表详情原文：列表页如果已经有足够摘要，作为最低保真输入。
2. 官方/移动端 API：优先读取 JSON 中的正文、长文、结构化字段。
3. HTML 主体抽取：引入可配置正文选择器，并用 Trafilatura、BeautifulSoup/lxml 做通用抽取。
4. 结构化数据抽取：解析 `application/ld+json`、OpenGraph、微博长文、新闻正文脚本变量。
5. 搜索补全：对热搜词、短标题类内容，抓取同源搜索结果前 N 条，合并高相关正文。
6. 浏览器渲染兜底：只对高价值且常规策略失败的内容启用，避免成本失控。
7. 低分重试：`failed/list_only/partial<60` 自动进入 `detail_job`，根据失败原因选择不同策略。

详情评分要从单一长度和标题相关性，升级为多维评分：

- 完整度：正文长度、段落数、是否包含时间/主体/动作/结果。
- 相关性：标题关键词、核心实体、来源 URL、搜索词与正文匹配。
- 可读性：去导航、去广告、去登录提示、去反爬壳页。
- 新鲜度：发布时间、抓取时间和事件更新时间。
- 可信度：来源权威、是否官方、是否多源印证。

#### 5.2.1 2026 年主流方案调研对比

Max 版本不建议押注单一爬虫框架，而是按页面类型选择工具。调研结论如下：

| 方案 | 适合场景 | 优点 | 风险/不足 | 结论 |
| --- | --- | --- | --- | --- |
| Scrapy | 大规模静态站点、分页、站内递归、规则化结构化数据 | 成熟稳定，内置调度、优先级队列、去重、中间件、AutoThrottle，适合长期维护 | 对当前项目侵入较大；与现有 APScheduler + SQLAlchemy 链路需要改造 | 不作为 Max 一期主框架，二期可用于高吞吐频道 |
| HTTPX + BeautifulSoup/lxml | API、RSS、移动端 JSON、普通 HTML 新闻页 | 轻量，容易接入现有 `BaseCrawler`；HTTPX 支持 async、HTTP/2、连接池和严格超时 | 需要自己维护队列、限流、重试和解析策略 | 作为列表采集和低成本详情抓取主路径 |
| Playwright | JS 渲染、登录态页面、动态详情、反爬壳页诊断 | 官方支持 Python，可驱动 Chromium/Firefox/WebKit，适合动态页面和复杂交互 | 成本高、并发低、资源占用大，不适合全量抓取 | 作为高价值失败详情的兜底策略 |
| Crawlee for Python | 需要统一 HTTP/BeautifulSoup/Parsel/Playwright crawler、request queue 的新爬虫 | 提供 RequestQueue、PlaywrightCrawler、BeautifulSoupCrawler 等封装，适合新建独立爬虫任务 | 项目当前已有自研调度和数据模型，引入后边界需谨慎 | 作为新渠道试点，不替换现有核心链路 |
| Crawl4AI | 面向 LLM/RAG 的网页转 Markdown、动态页面、结构化抽取 | 异步、浏览器驱动、自动 Markdown、支持 CSS/XPath/LLM 提取，适合详情页转干净文本 | 版本演进快；浏览器模式仍有成本；需要样本回放验证稳定性 | 作为详情补全实验策略，先用于高价值低分样本 |
| Trafilatura | 新闻、博客、文章正文抽取和元数据抽取 | 专注主文本、元数据、评论抽取，能减少导航/页脚噪声，Python 集成简单 | 对强 JS 页面无能为力；仍需源 HTML 质量 | 作为通用 HTML 正文抽取首选组件 |
| Firecrawl / Jina Reader API | 外部托管网页转 Markdown、搜索补全、临时应急 | 省基础设施，输出 LLM 友好文本，可处理部分动态内容 | 外部依赖、成本、稳定性、合规和私有化不可控 | 只作为离线调研和人工诊断工具，不进入默认生产链路 |

#### 5.2.2 最终选型

Max 一期选择“自研任务队列 + HTTPX + Trafilatura + Playwright/Crawl4AI 兜底”的混合方案：

1. 保留现有 `BaseCrawler` 和渠道 registry，避免一次性迁移到 Scrapy/Crawlee 造成大面积重构。
2. 把 `requests.Session` 逐步升级为统一 `FetchClient`，底层优先使用 HTTPX，支持同步兼容、异步并发、HTTP/2、连接池、超时、重试、代理和响应样本落盘。
3. 引入 Trafilatura 作为通用正文抽取器，优先处理新闻、博客、公告、文章页。
4. 为每个渠道配置 `DetailStrategyChain`，策略顺序为：列表原文、官方/移动端 API、结构化 JSON、Trafilatura、渠道 CSS/XPath、搜索补全、Playwright、Crawl4AI。
5. Playwright 仅处理 `value_score >= 70` 且常规策略失败的详情任务，并设置每日预算、单域名并发和超时。
6. Crawl4AI 作为实验性策略，用于需要 Markdown 输出、动态页面抽取和 AI 摘要前置清洗的详情页；稳定后再扩大范围。
7. Scrapy/Crawlee 暂不替换现有架构。若后续某类渠道需要深度递归或高吞吐，再以独立 worker 方式接入，并把结果写回统一 `Info` 和 `source_signal`。

#### 5.2.3 详情策略链设计

新增 `DetailStrategy` 接口：

- 输入：`info_id`、`channel_code`、`source_url`、列表标题、列表摘要、渠道画像、上次失败原因。
- 输出：正文、发布时间、作者/来源、图片、结构化元数据、策略名、耗时、失败原因、原始样本路径。
- 约束：每个策略必须可单测、可记录样本、可配置超时和最大正文长度。

策略链执行规则：

1. 先跑低成本策略，成功且 `detail_score >= 70` 立即停止。
2. 如果命中反爬、登录、JS 壳页，不重复跑同类 HTTP 策略，直接进入渲染或延迟重试。
3. 如果命中弱相关，优先搜索补全或同源候选页，而不是盲目保留长正文。
4. 如果正文短但来源权威，允许进入 `partial`，并创建补偿任务。
5. 所有失败都写入 `InfoAcquisitionLog` 和 `detail_job.last_failure_reason`，用于管理后台归因。

#### 5.2.4 组件落地清单

- `FetchClient`：统一 HTTPX 请求、连接池、限流、重试、响应大小限制、样本保存。
- `HtmlArticleExtractor`：封装 Trafilatura、BeautifulSoup/lxml、结构化数据解析。
- `RenderedPageFetcher`：封装 Playwright，支持页面等待、正文选择器、截图/HTML 样本保存。
- `MarkdownDetailExtractor`：封装 Crawl4AI，输出 Markdown 和过滤后的正文。
- `SearchSupplementStrategy`：对热搜词和短标题抓同源搜索候选，按相关性合并正文。
- `DetailStrategyChain`：按渠道画像编排策略，负责短路、降级和评分。
- `DetailReplayTest`：每个核心渠道保存真实样本，策略变更必须回放通过。

### 5.3 采集及时性优化

调度从固定间隔改为“基础轮询 + 热点加速 + 失败退避”：

- 基础轮询：每个渠道有默认 interval，并支持在管理后台手动配置。
- 热点加速：热搜、榜单、突发新闻类渠道在高峰时段提频到 3-5 分钟。
- 低效降频：连续反爬、403、空结果渠道自动降频并告警。
- 分类并行：同一分类内渠道并行执行，但遵守单渠道并发和域名限流。
- 增量游标：优先按发布时间、榜单页码、source_id 游标采集新增内容，避免重复扫全量。
- 事件驱动补抓：当某事件热度上升但详情低分时，自动提高相关 `detail_job` 优先级。

#### 5.3.1 管理后台手动配置采集间隔

为保证数据新鲜度，Max 版本必须把采集间隔从代码配置升级为管理后台配置：

- 渠道列表页增加“采集间隔”列，展示基础间隔、热点间隔、实际生效间隔、最后修改人和最后修改时间。
- 渠道编辑页支持配置 `base_interval_minutes`、`hot_interval_minutes`、`min_interval_minutes`、`max_interval_minutes`、是否启用热点加速、是否启用失败退避。
- 保存时做边界校验：热点渠道最小 3 分钟，普通渠道最小 10 分钟；如低于安全阈值，需要二次确认并记录审计日志。
- 调度器不再只读取 `config.py`，而是以数据库 `channel_profile` 为准；`config.py` 仅保留首次初始化默认值。
- 配置修改后写入 `schedule_version`，调度器每 30-60 秒检测版本变化并热更新任务，不要求重启服务。
- 连续失败时系统可临时退避，但管理后台仍保留人工覆盖入口；页面必须展示“人工配置值”和“系统退避后的生效值”的差异。
- 所有间隔修改写入审计日志，支持回看是谁在什么时间把某渠道从多少分钟改成多少分钟。

### 5.4 数据价值筛选

Max 版本需要区分“采到了”和“值得展示”。建议新增 `value_score`：

- 热度信号：榜单排名、热度值、互动量、转发评论、跨平台出现次数。
- 来源信号：官方媒体、垂直媒体、社区内容、个人观点按不同权重处理。
- 时效信号：越新权重越高，过期内容降权，但重大事件保留。
- 事件信号：同一核心实体在多渠道出现，事件分提高。
- 内容信号：具备时间、地点、人物、组织、数据指标、明确进展的内容优先。
- 用户信号：后续接入点击、收藏、停留、分享，反哺价值排序。

展示池只使用 `quality_score >= 60` 且 `value_score >= 50` 的内容；低质但可能重要的内容进入观察池等待补全，不直接污染事件流。

### 5.5 质量治理闭环

当前已有 `DataQualitySnapshot`，Max 版本要扩展为每日和每小时的治理闭环：

- 每小时生成渠道质量榜：成功率、空结果、详情完整率、低分样本、平均延迟。
- 每日生成问题归因：反爬、解析器失效、正文选择器失效、重复过高、语义缺失。
- 自动创建治理任务：详情失败率超过阈值时，生成渠道修复任务和样本。
- 管理后台增加“低分样本池”：支持按渠道、失败原因、评分区间查看原始样本。
- 质量门禁：事件重建时默认排除低分内容，除非来源权威或热度极高。

### 5.6 环境和迁移可靠性

Max 版本启动前必须做 schema 健康检查：

- 检查 ORM 模型字段与数据库字段是否一致。
- 检查 Pro/MySQL schema 是否完整应用到当前环境。
- 发现缺字段时输出明确迁移建议，不让调度器带病启动。
- SQLite 仅保留本地开发用途，主验证链路统一使用 MySQL。
- 增加迁移脚本幂等测试，覆盖 `event_key`、采集日志、质量快照等关键字段。

## 6. 阶段计划

### 阶段一：采集质量基线与问题样本库

目标：先量化问题，不盲目改爬虫。

- 建立 Max 质量指标口径：完整率、新鲜度、低质率、重复率、补偿恢复率。
- 扩展 `DataQualitySnapshot`，按渠道、分类、时间窗口统计。
- 增加低分样本池 API 和管理后台入口。
- 修复本地/测试 schema 健康检查，避免质量报告因旧库失败。
- 输出第一版渠道健康榜，确定 Top 5 优先治理渠道。

验收：管理后台可看到每个渠道近 24 小时质量画像，并能定位低分样本和失败原因。

### 阶段二：详情补全与重试队列

目标：显著提升详情完整率。

- 新增 `detail_job` 队列，承接 `pending/list_only/failed/partial<60` 内容。
- 统一详情策略链，抽离 HTTPX、Trafilatura、结构化数据、Playwright、Crawl4AI 等策略组件。
- 为微博、头条、知乎、36 氪、CSDN、掘金优先接入多策略补全。
- 为失败原因建立策略路由：反爬、弱相关、正文过短、壳页面、网络错误分别处理。
- 详情补全完成后自动重算质量分，并触发事件重建。

验收：核心展示池详情完整率达到 70% 以上，补偿恢复率达到 40% 以上。

### 阶段三：智能调度与及时性提升

目标：让热点数据更快入库。

- 新增 `crawl_job` 队列，支持优先级、状态、重试、下次执行时间。
- 新增管理后台采集间隔配置，调度器按 `channel_profile` 热更新生效。
- 分类内渠道并行执行，单渠道限流。
- 热点类渠道支持 3-5 分钟加速轮询。
- 连续失败渠道自动退避，避免阻塞全局采集。
- 增量游标减少重复采集成本。

验收：热点类内容入库 P90 延迟 <= 10 分钟，非热点核心渠道 P90 <= 30 分钟。

### 阶段四：渠道扩容与价值评分

目标：补齐“最新、最热、最有价值”的来源网络。

- 新增百度热搜、财联社、机器之心、InfoQ、GitHub Trending、AP/BBC 等第一批 Max 渠道。
- 建立 `source_signal`，保存榜单排名、热度、互动量、发布时间等信号。
- 新增 `value_score`，用于事件排序和展示池筛选。
- 事件聚合引入跨源扩散权重，减少单一来源噪声。

验收：核心分类每类至少 4 个稳定渠道；事件排序能体现热度、新鲜度和多源验证。

### 阶段五：运营化监控与自动治理

目标：让采集问题可发现、可归因、可修复。

- 管理后台增加渠道健康页、详情补偿队列页、低分样本页。
- 增加告警规则：渠道失败率、低分率、新鲜度延迟、空结果异常。
- 每日自动生成采集质量报告，列出优先修复渠道和样本。
- 完善端到端测试：真实样本回放、详情策略单测、调度队列集成测试。

验收：出现采集质量下降时，系统能在 1 小时内暴露问题渠道、失败原因和样本。

## 7. 技术任务拆解

### 后端与数据库

- 新增 `channel_profile/crawl_job/detail_job/content_quality_score/source_signal` 表。
- `channel_profile` 增加基础间隔、热点间隔、最小/最大间隔、人工覆盖、实际生效间隔和配置版本字段。
- 增加 Alembic 或等价迁移管理，补齐启动前 schema 校验。
- 抽象 `DetailStrategy` 接口，统一策略输入输出。
- 抽象 `CrawlerCapability`，让渠道声明支持的列表源、详情源和限流配置。
- 重构 scheduler：从直接按分类执行改为任务队列执行器。
- 扩展事件重建：引入 `quality_score/value_score/freshness_score`。

### 爬虫与解析

- 为每个核心渠道补齐列表源、详情源、备用源。
- 引入 HTTPX、Trafilatura、Playwright、Crawl4AI、BeautifulSoup/lxml 和结构化数据解析。
- 增加反爬失败识别、响应样本保存、选择器失效检测。
- 增加真实样本回放测试，避免线上页面变动后无感失败。

### 管理后台

- 渠道健康榜：成功率、平均延迟、详情完整率、低分率。
- 渠道配置页：支持手动调整采集间隔、热点加速间隔和失败退避开关。
- 详情补偿队列：待处理、重试中、失败、已恢复。
- 低分样本池：原始内容、失败原因、命中规则、策略链。
- 质量趋势：按小时/天展示完整率、新鲜度和重复率。

### 前端事件消费

- 事件流默认过滤低质内容。
- 事件详情展示代表性来源时优先选择高质量、高权威来源。
- 对低详情分事件展示“持续更新中”，避免把短摘要包装成完整解读。

## 8. 测试与验收方案

- 单元测试：详情策略、质量评分、重复判断、价值评分、schema 校验。
- 集成测试：列表采集入库、详情补偿、事件重建、质量快照。
- 样本回放：每个核心渠道保存 20 条真实 HTML/JSON 样本，解析规则变更必须回放通过。
- 调度配置测试：管理后台修改采集间隔后，调度器在 60 秒内热更新，且审计日志完整记录。
- 压力测试：并行采集、队列重试、限流退避、数据库写入吞吐。
- 线上试运行：至少连续 72 小时观察渠道成功率、详情完整率、新鲜度和告警准确率。

## 10. 优先级建议

第一优先级不是新增更多渠道，而是把“质量指标、详情补偿、智能调度”打牢。否则渠道越多，低分数据越多，事件流反而更乱。

推荐顺序：

1. 先做质量基线和 schema 健康检查。
2. 再做详情补偿队列和通用详情策略。
3. 然后做智能调度和热点加速。
4. 最后做渠道扩容和价值评分。

这样 Max 版本可以每一阶段都产出可验证收益，而不是等到大重构完成后才知道采集质量是否真的变好。
