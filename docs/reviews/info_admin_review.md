# info-admin 工程 Review 总结

## 1. 工程定位

**info-admin** 是整个 info-ai 体系的**运营管理后台**，面向内部运营人员和系统管理员，提供数据采集监控、渠道管理、数据质量治理、审计日志查询等能力。

核心功能模块：
- **Dashboard**：采集渠道数、热点事件数、原始内容数、新鲜度风险渠道数
- **采集运行**：查看各渠道最近采集批次的状态、数量、耗时
- **数据质量**：重复标题、正文缺失、低详情评分等质量指标
- **详情任务**：管理低质量/失败内容的二次抓取任务（重试/取消）
- **渠道/分类管理**：增删改查渠道和分类配置
- **审计日志**：管理操作全量记录

## 2. 系统架构

### 2.1 技术栈
- **Framework**: Vue 3.5 + Composition API
- **Router**: Vue Router 4（history 模式）
- **Build Tool**: Vite 6
- **Language**: TypeScript
- **Testing**: Vitest + happy-dom
- **State Management**: 无 Pinia/Vuex
- **HTTP Client**: 原生 `fetch` 封装
- **UI Library**: 无（纯手写 CSS）
- **Auth**: JWT + 路由守卫

### 2.2 模块结构
```

├── views/              # 页面
│   ├── LoginView.vue
│   ├── DashboardView.vue
│   ├── CrawlRunsView.vue
```

### 2.3 架构评价
**优点**：
- 路由守卫实现简洁，`requireAdminAuth` 统一校验登录态
- 布局抽象为 `AdminLayout`，子页面通过嵌套路由渲染，结构清晰
- `MetricCard` / `DataPanel` 等组件抽象合理，Dashboard 信息密度高
- 测试覆盖登录和路由守卫

**问题**：
- **无全局状态管理**：`authStore.ts` 仅做 token 存储，业务数据全在页面内管理
- **无图表库**：数据质量、采集趋势等数据未可视化，纯数字展示难以发现趋势
- **无表格组件**：渠道列表、审计日志等用原生 HTML 表格，排序/筛选/分页功能缺失
- **无表单校验库**：新增/编辑渠道表单无统一校验逻辑

## 3. 代码逻辑

### 3.1 Dashboard (`DashboardView.vue`)
- 同时请求 `overview` 和 `channelHealth`，使用 `Promise.all` 并行
- `staleChannels` 计算属性筛选 6 小时未更新的渠道，按健康分排序
- **问题**：时间解析使用 `replace(' ', 'T')` 手动处理，未使用日期库（如 dayjs）

### 3.2 路由守卫 (`router/guards.ts`)
- `requireAdminAuth` 检查 `localStorage` 中的 token，无 token 跳转登录页
- **问题**：未校验 token 是否过期，过期 token 会带入 API 请求导致 401
- **问题**：未做权限细粒度控制（仅校验登录，未校验角色）

### 3.3 API 层 (`services/adminApi.ts`)
- 封装了管理后台所需的全部 API
- 但 `httpClient.ts` 中未统一处理 401，各页面自行处理错误

### 3.4 样式 (`styles/base.css`)
- 自定义 CSS 变量定义了管理后台的设计系统
- 但缺少暗色模式、响应式断点
- 管理后台在移动端几乎不可用（但管理后台通常 PC 使用，可接受）

## 4. 页面布局与视觉

### 4.1 布局特点
- 侧边栏导航 + 内容区两栏布局（预期，但需确认 `AdminLayout.vue` 实现）
- Dashboard 采用 metric 卡片 + 数据面板网格
- 页面标题使用 `eyebrow`（小写英文标签）+ 中文大标题的双层结构

### 4.2 视觉风格
- 与 info-max 保持一致：蓝色主色调 + 浅灰背景
- 状态标签（`StatusBadge`）使用彩色圆点 + 文字，清晰直观
- 空状态（`EmptyState`）统一处理，避免裸数据缺失

### 4.3 UI 问题
- **缺少数据可视化**：采集趋势、质量变化等时间序列数据无图表
- **表格功能简陋**：无排序、无筛选、无分页、无批量操作
- **缺少操作反馈**：重建事件、刷新质量等耗时操作无进度提示
- **面包屑缺失**：深层页面（如详情任务）无返回路径指示

## 5. 产品定位分析

info-admin 的定位是**"运维监控与治理工具"**，服务于内部团队而非 C 端用户。

**当前匹配度**：
- 采集监控：Dashboard 核心指标一目了然
- 质量治理：重复、缺失、低质量数据可发现
- 任务管理：详情补偿任务可重试/取消

**缺失的运维能力**：
- 无实时告警（渠道失败、质量骤降时无通知）
- 无操作审批（重建事件等高风险操作无二次确认/审批流）
- 无配置版本管理（渠道配置修改无回滚能力）
- 无多租户/多角色（管理员/运营/只读角色未区分）

## 6. 问题汇总与优化方向

### 6.1 高优先级（近期）

| 问题 | 影响 | 优化方向 |
|------|------|----------|
| 无数据可视化 | 趋势难以发现，监控价值低 | 引入轻量图表库（如 Chart.js 或 ECharts） |
| 表格功能缺失 | 数据量大时无法高效浏览 | 封装表格组件：排序、筛选、分页、批量操作 |
| 无 Token 过期处理 | 用户工作到一半被踢出，数据丢失 | 路由守卫增加 token 过期校验，提前弹窗提示 |
| 耗时操作无反馈 | 用户重复点击重建事件 | 增加 Loading + 进度轮询 + Toast 通知 |
| 无表单校验 | 非法输入直接提交到后端 | 增加统一表单校验（如 VeeValidate） |

### 6.2 中优先级（中期）

| 问题 | 影响 | 优化方向 |
|------|------|----------|
| 无实时告警 | 渠道挂掉几小时才发现 | 接入 WebSocket 或轮询告警，异常时弹窗/邮件通知 |
| 无操作审批 | 高风险操作误触后果严重 | 重建事件等操作增加二次确认 + 审批流 |
| 无配置版本管理 | 配置改错无法回滚 | 渠道/分类配置增加修改历史和回滚能力 |
| 无多角色权限 | 所有人都是管理员 | 引入 RBAC：admin / operator / viewer |
| 缺少导出能力 | 审计日志、质量报表无法导出 | 增加 CSV/Excel 导出功能 |

### 6.3 低优先级（长期）

| 问题 | 影响 | 优化方向 |
|------|------|----------|
| 无暗黑模式 | 夜间运维体验差 | CSS 变量支持暗色主题 |
| 纯手写 CSS 效率低 | 新页面开发慢 | 引入轻量 UI 框架（如 Ant Design Vue 或 Element Plus） |
| 无响应式 | 紧急时无法手机查看 | 增加基础移动端适配 |
| 缺少 APM 接入 | 前端性能问题难定位 | 接入 Sentry 或自建前端监控 |

---

## 7. 总体评分

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构设计 | 6.5/10 | 分层清晰，但缺少状态管理和图表库 |
| 代码质量 | 6.5/10 | 逻辑清晰，测试有覆盖，但日期处理粗糙 |
| UI/UX 设计 | 5.5/10 | 信息展示完整，但无图表、无表格功能、无反馈 |
| 运维能力 | 6/10 | 监控指标有，但缺少告警、审批、版本管理 |
| 业务匹配 | 6.5/10 | 核心运维功能具备，但高级治理能力缺失 |

**一句话总结**：info-admin 是一个"功能完整但体验朴素"的管理后台，能满足日常运维需求，但**数据可视化**、**表格功能**和**实时告警**是急需补齐的短板，否则运营团队难以高效发现和处理问题。
