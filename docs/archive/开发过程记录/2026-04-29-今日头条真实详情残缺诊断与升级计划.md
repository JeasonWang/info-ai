# 今日头条真实详情残缺诊断与升级计划

日期：2026-04-29

## 结论

今日头条“立即采集”后详情大量残缺，主要不是数据库保存截断，也不是列表接口返回数量过少，而是当前详情抓取链路没有拿到真实正文。

真实热榜接口可以稳定返回热点主题列表，但字段主要是标题、热度、标签、话题页 URL、图片和分类；它不是文章详情接口。当前网页兜底访问 `toutiao.com/trending/...` 或部分 `toutiao.com/article/...` 时，多数只得到 JavaScript 壳页，内容类似“今日头条 您需要允许该网站执行 JavaScript”。搜索接口在真实请求下也经常返回空列表，无法补齐正文。

另外，之前系统模拟 seed 数据的详情是完整的，且 `detail_strategy=seed`、`detail_score=100`，这会掩盖真实渠道质量问题。分析今日头条质量时必须排除 seed 数据，只看 `channel_code=toutiao` 且非 seed 策略的数据。

## 真实数据证据

### 真实热榜接口

请求源：

```text
https://www.toutiao.com/hot-event/hot-board/?origin=toutiao_pc
```

实际返回 50 条热榜主题，字段包含：

- `ClusterId`
- `Title`
- `Label`
- `QueryWord`
- `Url`
- `HotValue`
- `Image`
- `InterestCategory`

未发现稳定正文类字段。部分响应会有 `HotDesc` 或 `Abstract`，但真实请求中大量为空或很短。

### 真实详情复测

2026-04-29 复测前 5 条热榜：

| 标题 | 列表长度 | 详情状态 | 策略 | 评分 | 失败原因 |
| --- | ---: | --- | --- | ---: | --- |
| 榴莲价格大降 | 6 | failed | web_fallback | 0 | shell_page |
| 中国军号：日本打不起“长期战争” | 20 | list_only | list_fallback | 10 | detail_unavailable |
| 稳就业在行动 | 6 | failed |  | 0 | detail_unavailable |
| 小车司机拔掉等红灯摩托车钥匙后逃离 | 17 | failed | web_fallback | 0 | shell_page |
| 塔克拉玛干沙漠边28万只河蟹入塘 | 16 | failed | web_fallback | 0 | shell_page |

这说明当前链路抓到的是热点主题，不是可分析的完整事件详情。

### 入库质量统计

本地 MySQL `info-max` 中今日头条真实数据曾出现：

| 状态 | 策略 | 数量 | 平均分 | 平均正文长度 |
| --- | --- | ---: | ---: | ---: |
| partial | hot_board_detail | 49 | 61.9 | 24.2 |
| list_only | list_fallback | 9 | 10.0 | 14.7 |
| complete | seed | 2 | 100.0 | 175.5 |

其中 `complete/seed` 是模拟数据，不代表真实采集能力。`partial/hot_board_detail` 平均长度只有 24.2，属于标题、标签级内容，不应被当成可用详情。

## 已完成修正

1. 今日头条列表采集保留真实热榜字段：
   - `Url` 写入 `source_url`
   - `QueryWord` 写入 `_query_word`
   - `Image` 写入 `_image_url`
   - `InterestCategory` 写入 `_interest_category`

2. 今日头条搜索补偿优先使用 `QueryWord`，比只用标题更接近头条自身热榜查询语义。

3. 热榜详情候选增加质量门槛：只有标题、标签、`hot` 这类短字段时，不再生成 `hot_board_detail` 候选，避免“标题+标签”被误评为 `partial`。

4. 增加回归测试：
   - 保留真实热榜 URL 和查询词。
   - 拒绝只有标题和标签的伪详情。
   - 保持 JS 壳页识别为失败，不进入详情内容。

5. 数据质量报告新增真实口径：
   - `seed_detail_count`：模拟详情数量。
   - `real_detail_total`：非 seed 真实采集数量。
   - `real_complete_detail_count`：非 seed 且达到完整详情标准的数量。
   - `real_complete_detail_ratio`：真实详情完整率。

6. 管理后台“数据质量”快照列表展示真实完整率、真实完整数和 seed 数量，避免运营验收时被模拟数据误导。

## 根因拆解

### 不是访问网址数据少

热榜入口本身可以拿到 50 条热点，当前代码只取前 20 条是产品策略，不是“只有几条数据”的根因。数据条数可以扩展，但扩展后仍然只是热点主题。

### 是详情没有爬取完整

详情残缺来自三层失败：

1. 热榜 API 不提供正文，只提供热点主题元数据。
2. 搜索 API 对真实请求经常不返回正文数据。
3. 页面 HTML 依赖前端 JavaScript 和动态接口，普通 HTTP 抓取只能拿到壳页。

### 质量评分存在误判

短内容包含标题时，通用详情评分会给到 60 分以上，导致 `hot_board_detail` 短文本变成 `partial`。今日头条这种渠道必须做源头级质量门槛，不能把热榜元数据当详情正文。

## 后续升级计划

### P0：真实质量校准

- 排除 `detail_strategy=seed` 的质量统计口径。
- 今日头条短文本默认进入 `list_only/failed`，进入详情补偿队列。
- 后台列表增加按渠道查看真实详情完整率，重点展示 `complete_non_seed_rate`。

### P1：动态详情抓取

- 引入浏览器渲染型详情策略，优先调研并接入 Playwright/Crawl4AI。
- 对 `toutiao.com/trending/...` 执行页面渲染，监听 XHR/Fetch 响应，提取话题下相关文章列表和正文片段。
- 对 `toutiao.com/article/...` 执行渲染后抽取正文，并识别登录、验证码、频控、JS 壳页。
- 把动态抓取作为成本较高的补偿策略，只处理高热度、低详情分、进入事件池候选的数据。

### P2：多源交叉补全

- 使用今日头条 `QueryWord` 作为事件查询词，到其他新闻源补齐事实详情。
- 同一事件下保留多来源证据，不把其他来源正文伪装成头条原文。
- 事件详情页展示“头条热度 + 多源详情”的组合质量。

### P3：反爬与时效治理

- 增加渠道级请求限速、失败退避、User-Agent 池、Cookie 池配置。
- 对 `shell_page`、`anti_crawl_blocked`、`detail_unavailable` 分开统计。
- 采集任务按热度、发布时间、详情分动态排序，高价值低详情内容优先补偿。

## 验收标准

- 今日头条真实数据中 seed 数据不参与完整率统计。
- “标题+标签”不再被标记为 `partial/hot_board_detail`。
- 低质量真实头条数据自动进入详情补偿队列。
- 浏览器渲染策略接入后，今日头条高热榜前 20 条中至少 60% 能拿到 80 字以上可读详情，且壳页内容不入库。
