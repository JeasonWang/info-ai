## 执行动作

1. 使用本地 MySQL `127.0.0.1:3306/info-max`。
2. 清空历史采集、详情任务、事件、质量快照和运行日志数据。
3. 全渠道重新采集真实线上数据。
4. 对新入库数据执行详情页抓取和低质量详情补偿。
5. 重建事件与质量快照。
6. 从 MySQL 汇总每个渠道详情完整性，并抽查真实正文样本。

## 本次代码修复点

- 清洗层支持 `_allow_title_only`：微博、小红书、博客园、36氪、CSDN、掘金等“列表入口短、详情页长”的渠道不再被清洗阶段误删，先入库再抓详情。
- `source_url` 增加长度保护：超长 URL 先去掉 query/fragment，仍超长则截断到 500 字符，避免真实热榜 URL 写入 MySQL 失败。
- 今日头条热榜 URL 规范化为短链接，保留原始热榜 URL 到内部字段，解决 `source_url` 过长问题。
- 知乎详情链路支持轻量候选短路：`search_embedded` / `answer_api` 已达到可用状态时不再触发浏览器渲染。
- 微博详情链路支持轻量候选短路：话题搜索或热榜上下文可用时不再继续访问当前环境稳定失败的移动端搜索。
- 博客园详情页改为单次 HTML 获取后复用 `cnblogs_post_body` 与通用正文抽取，减少重复请求和超时放大。
- 掘金详情 API 对 `data = null` 做防御处理，避免无效异常日志。

## 后续重点

1. Reuters 当前入口在本地网络环境返回 401/404，需要替换为可稳定访问的授权源、RSS 镜像或其他国际新闻源。
2. 微博正文仍偏短，后续需要增加微博登录态 Cookie、移动端接口降级熔断和更稳定的话题页正文抽取。
3. 小红书详情成功率受 `window.__INITIAL_STATE__` 是否含完整笔记影响，后续需要接入浏览器渲染或登录态。
4. 掘金主采集过快时详情会退化为列表摘要，补偿队列可恢复；后续应为掘金详情加入限速和稳定 API 优先级。
5. 今日头条依赖浏览器渲染，后续要复用 browser/context，降低 20 条采集的启动成本。

## 2026-05-02 追加优化

本轮继续针对弱渠道做了专项优化，并完成 `148 passed` 回归验证。

- Reuters：在官方 API、RSS、网页入口均返回 401/404 或不可访问时，增加 Google News Reuters 索引兜底。该策略只恢复真实 Reuters 新闻线索和索引摘要，不伪造 Reuters 正文；索引摘要进入 `news_index_summary`，质量状态至少可进入 partial。
- 微博：热榜上下文补充热榜标签、分类、排名、热度值，避免只有标题时详情过短。线上轻量验证中，微博样本从约 40 字提升到约 70 字，详情分数从约 58 提升到 67/68。
- 小红书：列表页解析保留 `noteCard.desc`，并把较完整的列表摘要作为 `_search_content`；详情页改为单次 HTML 获取后复用 `INITIAL_STATE` 和 HTML 文本抽取，减少重复请求。线上轻量验证中，样本详情可达 511 字。
- 掘金：网页详情增加内联状态 `mark_content/content` 抽取，并统一使用通用 HTML 正文抽取器，避免壳页面文本污染。线上轻量验证中，前两条详情分别获取 2055 字和 4028 字完整正文。
- 今日头条测试：在壳页面过滤测试中显式关闭渲染器，避免本地 Chromium 存在时测试误访问真实页面导致波动。
