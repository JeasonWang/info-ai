# 数据库字段注释规范

## 目标

后续所有数据库字段都必须带中文注释，保证产品、开发和数据分析时能快速理解字段含义，避免字段越加越多但没人敢用。

## 规则

- SQLAlchemy 模型新增 `Column` 时必须填写 `comment="中文说明"`。
- 注释要说明业务含义，不只重复字段名。
- 状态类字段要写清楚枚举含义，例如 `pending/list_only/partial/complete/failed`。
- 派生字段要说明来源，例如由采集器写入、由语义解析生成、由事件重建生成。
- 不确定是否长期保留的字段不要先入库，优先在采集日志或质量报告中观察。

## 当前自动校验

后端测试 `test_database_field_comments.py` 会扫描所有 ORM 表字段。如果新增字段没有注释，测试会失败。

## Plus 收尾新增质量体检

新增管理接口：

```text
GET /api/admin/data-quality-report
```

该接口会返回：

- `info.low_quality_count`：低质量原始内容数量。
- `info.title_content_duplicate_count`：标题和正文重复数量。
- `info.incomplete_detail_count`：详情采集不完整数量。
- `info.semantic_scope_total`：需要统计科技语义字段的内容数量。
- `info.missing_semantic_count`：科技/AI 范围内关键实体、关键词、科技主题缺失数量。
- `info.duplicate_title_count`：标题重复数量。
- `info.duplicate_source_url_count`：来源 URL 重复数量。
- `event.title_summary_duplicate_count`：事件标题和一句话摘要重复数量。
- `samples.incomplete_details`：详情不完整样本，用于定位需要补强的采集渠道。
- `samples.missing_semantics`：科技语义缺失样本，用于补充解析词典或判断是否不该展示该字段。
- `recommendations`：下一步治理建议。

这个接口不修改数据，只负责暴露问题，方便我们判断 Plus 版本是否具备收尾条件。

## Plus 收尾维护接口

```text
POST /api/admin/archive-low-quality
```

软删除明显低质量内容，例如标题和正文重复、详情只停留在列表摘要的记录。该接口会重建事件流。

当前会识别的不可用详情包括：

- 页面要求允许 JavaScript 才能查看。
- 登录、验证、访问频次过高等反爬拦截页。
- 标题和正文高度重复的伪详情。

```text
POST /api/admin/archive-duplicate-titles
```

软删除重复标题内容，每组保留详情质量最高的一条。该接口会重建事件流。

以上两个接口都是软删除，只设置 `info.is_deleted=1`，不物理删除数据库记录。
