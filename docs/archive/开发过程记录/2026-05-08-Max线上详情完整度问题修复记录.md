# Max 线上详情完整度问题修复记录

## 背景

Max 版本上线后，线上观察到部分 `info-mvp` 事件详情页正文明显短于原始站点文章，容易出现“详情页只展示了开头一小段”的体验。排查链路覆盖采集清洗、详情质量判定、事件摘要构建、`info-serve` 事件详情 API 和 `info-mvp` 页面展示。

## 原因定位

1. 采集清洗层 `clean_content()` 对列表内容做了 500 字硬截断。部分渠道在列表阶段已经拿到较完整正文时，会被提前截断。
2. 详情补偿队列只按 `detail_score < 60` 或正文长度小于 30 字判断，文章类渠道即使只有几十到几百字，也可能不再进入补偿。
3. 事件详情页展示的是事件摘要，不是原始正文。事件摘要构建使用 120/180 字片段，`info-serve` 的多源视角也只返回 120 字预览，导致用户看到的内容像“异常截断”。
4. `info-mvp` 事件详情页没有把代表来源正文直接展示出来，用户必须再点“相关报道”进入单条信息详情才能判断底层采集是否完整。

## 本次修复

1. 采集清洗取消 500 字硬截断，改为 `CRAWLER_MAX_CONTENT_LENGTH` 异常保护上限，默认 12000 字。
2. 详情补偿队列改为按渠道完整度阈值判断。文章类渠道如 CSDN、掘金、博客园、36氪、Reuters 等，正文长度低于渠道 `complete_min_length` 会重新入队补偿。
3. 事件摘要构建改为按句子边界截断，避免无提示地切断半句话。
4. `info-serve` 的事件详情 API 在 `representative_sources` 中返回代表来源正文、详情状态、评分和正文长度，并优先选择完整度更高的来源。
5. `info-mvp` 事件详情页新增“代表原文”模块，默认展示前 800 字并支持展开完整正文；摘要区继续承担“快速看懂事件”的职责。

## 部署注意

容器环境已在 `.env.example`、`docker-compose.yml`、`deploy.sh` 中补充 `CRAWLER_MAX_CONTENT_LENGTH=12000`。线上更新后建议在管理后台“数据质量”页面执行“重抓低完整详情”，或调用 `POST /api/admin/retry-low-quality-details?limit=50`，让历史短正文数据重新进入详情补偿队列。

## 验收重点

1. 打开 `info-mvp` 事件详情页，确认“发生了什么”是摘要，“代表原文”展示真实来源正文。
2. 对文章型渠道的短正文数据，后台质量报告应能识别为详情不完整，并进入补偿队列。
3. 单条信息详情页应能展示完整正文，不再被采集清洗阶段压到 500 字以内。
