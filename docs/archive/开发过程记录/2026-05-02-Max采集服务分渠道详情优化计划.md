# Max 采集服务分渠道详情优化计划

日期：2026-05-02

## 目标

采集服务是信息达人 Max 版本的数据底座。本阶段目标不是简单扩大采集数量，而是让每个渠道稳定采集到完整、有价值、可分析的真实线上数据，尤其是正文详情必须能解释来源、质量和失败原因。

## 核心原则

1. 分渠道定义“完整”
   - 热搜/话题渠道需要事件背景、争议点、热度上下文。
   - 技术文章渠道需要正文主体、技术实体、方案/结论。
   - 财经指标渠道需要指标值、变化方向、市场解释和相关新闻。
   - 体育新闻渠道需要赛事背景、过程、结果和关键人物。

2. 真实线上数据验收
   - 每个渠道优化后必须跑真实采集。
   - 验收不看模拟数据，只看线上样本的 `detail_fetch_status/detail_score/detail_strategy/detail_content_length/content`。
   - 每轮至少抽样 5 条成功样本、5 条失败或低分样本。

3. 宁缺毋滥
   - 标题扩写、导航壳页面、登录页、验证码页、JS 空壳页不能进入事件分析主流程。
   - 详情失败可以入补偿队列，但不能伪装成完整正文。

## 渠道优先级

### P0：详情评分与质量诊断底座

涉及文件：

- `info_aggregation/services/detail_pipeline.py`
- `info_aggregation/services/data_quality_report.py`
- `info_aggregation/services/detail_jobs.py`
- `info_aggregation/scheduler/__init__.py`

优化内容：

1. 建立 `ChannelDetailProfile`
   - `channel_code`
   - `content_type`: `hot_topic/article/finance_indicator/social_note/sports_news`
   - `complete_min_length`
   - `partial_min_length`
   - `strong_complete_min_length`
   - `require_title_relevance`
   - `paragraph_min_count`
   - `noise_patterns`

2. 升级详情评分
   - 热搜类：80 字以上可完整，但必须有背景或上下文。
   - 技术文章类：300 字以下不轻易判完整，800 字以上且相关性好才高分。
   - 财经指标类：允许较短，但必须包含指标主体和解释性内容。
   - 体育新闻类：正文需要包含比赛/球队/人物/结果等信息密度。

3. 采集质量报表增强
   - 按渠道展示完整率、平均正文长度、低分数量。
   - 展示详情策略分布。
   - 展示失败原因 TopN。
   - 输出低质量样本，方便快速定位入口问题还是详情问题。

验收标准：

- 所有历史测试通过。
- 新增测试覆盖至少 4 类渠道 profile。
- 同样一段 80 字内容，在热搜渠道可 partial/complete，在技术文章渠道不能误判为 complete。

### P1：第一批弱渠道详情重构

优先渠道：博客园、路透、东方财富。

原因：

- 这三个渠道以文章或财经分析为主，价值高。
- 目前详情逻辑偏简单，容易出现摘要当正文或网页噪声当正文。
- 不强依赖登录态，适合作为统一详情能力的第一批落地渠道。

#### 博客园 `cnblogs`

目标：

- 从列表标题升级为完整技术文章正文。
- 优先提取 `cnblogs_post_body/cnblogs-post-body/article_content`。
- 保留代码块周围解释文字，但过滤导航、评论、版权和推荐区。

完整标准：

- `complete_min_length`: 500
- `partial_min_length`: 180
- 必须包含标题关键词或技术关键词。

验收：

- 真实采集 10 条，至少 6 条达到 complete 或高分 partial。
- 抽样正文应包含技术论述，而不是首页导航。

#### 路透 `reuters`

目标：

- 优先走 Reuters article API。
- API 失败时，从页面 JSON-LD、Next 数据或正文区块提取。
- 不能把 cookie 提示、订阅提示、导航区当正文。

完整标准：

- `complete_min_length`: 500
- `partial_min_length`: 180
- 英文新闻允许标题相关性用英文 token 判断。

验收：

- 真实采集 10 条，至少 5 条正文长度超过 500。
- 失败样本应有明确 `anti_crawl_blocked/shell_page/detail_unavailable` 原因。

#### 东方财富 `eastmoney`

目标：

- 指标数据本身要完整，同时补充市场解释。
- 搜索相关新闻时不只取第一条，合并 2-3 条高相关摘要。
- 指标类详情应包含：当前值、涨跌、时间、相关市场解释。

完整标准：

- `complete_min_length`: 160
- `partial_min_length`: 80
- 必须包含 `indicator_name` 或同义关键词。

验收：

- 国际金价、国际原油、美元人民币汇率三类都有详情。
- 每条都能解释指标变化，不只是行情数值。

### P2：第二批动态/反爬渠道

优先渠道：小红书、知乎、今日头条、微博。

原因：

- 数据价值高，但反爬和动态渲染强。
- 已经有一定优化基础，需要继续用真实失败样本迭代。

#### 小红书 `xiaohongshu`

目标：

- 增加 cookie 配置和 Playwright 渲染兜底。
- 从页面状态、DOM、渲染文本多策略提取笔记标题、正文、标签、互动数据。

完整标准：

- `complete_min_length`: 120
- `partial_min_length`: 50
- 必须不是备案页、登录页、空壳页。

#### 知乎 `zhihu`

目标：

- 继续优化热搜词到问题/文章/回答的映射。
- 对回答 API、搜索嵌入、渲染页分别记录策略质量。
- 针对 403、签名失效、登录态失效输出明确失败原因。

完整标准：

- 问答类 `complete_min_length`: 220
- 文章类 `complete_min_length`: 500
- 高赞回答组合优先。

#### 今日头条 `toutiao`

目标：

- 继续减少 Playwright 成本。
- 更准确区分热点事件页、文章页、搜索结果页。
- 对只有热榜摘要的内容标记为 partial，不冒充完整正文。

完整标准：

- 热点事件 `complete_min_length`: 120
- 文章详情 `complete_min_length`: 300

#### 微博 `weibo`

目标：

- 话题搜索、移动端搜索、热榜上下文继续保留。
- 增加对长微博、转发链、官方媒体账号结果的优先级。

完整标准：

- `complete_min_length`: 120
- 允许多条微博合并形成事件上下文。

### P3：第三批垂直渠道与持续治理

渠道：CSDN、掘金、36氪、央视体育、新浪体育。

#### CSDN / 掘金 / 36氪

当前基础较好，下一步是统一 profile 评分和真实样本复核。

完整标准：

- CSDN、掘金：`complete_min_length`: 600
- 36氪：`complete_min_length`: 500
- 低于 300 字通常只能 partial。

#### 央视体育 / 新浪体育

目标：

- 提取比赛新闻正文。
- 过滤彩票、图集、视频壳、导航专题。

完整标准：

- `complete_min_length`: 240
- 必须包含赛事、队伍、人物、比分、结果、赛程等至少一类实体信息。

## 实施步骤

### 阶段一：详情质量底座

1. 新增渠道 profile。
2. `run_detail_pipeline()` 支持 `channel_code/content_type/profile`。
3. 调度保存和详情补偿传入 channel code。
4. 更新测试，确保不同渠道同文本判定不同。

### 阶段二：第一批弱渠道重构

1. 博客园改成 `resolve_detail()` 多策略。
2. 路透改成 `resolve_detail()` 多策略。
3. 东方财富改成指标型 profile，并合并相关新闻上下文。
4. 对三渠道跑真实采集并记录样本。

### 阶段三：动态渠道增强

1. 小红书增加 cookie/渲染兜底。
2. 知乎增加失败原因诊断。
3. 头条继续优化热点事件页和文章页区分。
4. 微博增加官方媒体和长文优先级。

### 阶段四：报表和验收自动化

1. 增加采集质量 CLI 或 API。
2. 支持按渠道输出真实样本。
3. 每次优化后生成验收记录。

## 线上验收方法

每个渠道优化后执行：

```bash
cd info_aggregation
DB_TYPE=mysql DB_HOST=127.0.0.1 DB_PORT=3306 DB_USER=root DB_PASSWORD=root1234 DB_NAME=info-max ./.venv/bin/python -c "from main import register_all_crawlers; from crawlers.registry import crawler_registry; register_all_crawlers(); c=crawler_registry.get('CHANNEL_CODE'); items=c.safe_crawl(); print(len(items)); print(items[:3])"
```

然后触发详情：

```bash
cd info_aggregation
DB_TYPE=mysql DB_HOST=127.0.0.1 DB_PORT=3306 DB_USER=root DB_PASSWORD=root1234 DB_NAME=info-max ./.venv/bin/python -c "from scheduler import crawl_by_category; crawl_by_category('CATEGORY_NAME')"
```

验收 SQL：

```sql
select
  c.code,
  i.title,
  i.detail_fetch_status,
  i.detail_strategy,
  i.detail_score,
  i.detail_content_length,
  left(i.content, 500) as sample
from info i
join channel c on i.channel_id = c.id
where c.code = 'CHANNEL_CODE'
order by i.created_at desc
limit 10;
```

## 2026-05-02 第一阶段推进记录
