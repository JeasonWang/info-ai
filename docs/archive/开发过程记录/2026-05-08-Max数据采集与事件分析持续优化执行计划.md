# Max 数据采集与事件分析持续优化执行计划

## 目标

信息达人的核心竞争力是“完整、有价值、可分析”的热点事件数据。后续优化不再以单个爬虫临时修补为主，而是建设一套质量驱动的数据采集和事件分析底座，让系统能持续识别低质量数据、自动补偿详情、生成更有解释力的事件分析。

## 一、采集质量底座

1. 建立统一采集质量画像，输出正文完整度、信息价值、时效性、可信来源、风险原因和补偿建议。
2. 按渠道定义完整度规则。文章型渠道要求完整正文，社交型渠道要求话题背景和互动语境，问答型渠道要求问题与代表回答，财经体育新闻型渠道要求事实、指标或结果。
3. 所有质量判断统一从质量画像读取，避免 `detail_jobs`、质量报告、事件构建各自使用不同阈值。

## 二、详情补偿闭环

1. 对 `failed`、`list_only`、`partial`、短正文、低评分、反爬壳页面自动入队。
2. 补偿任务按失败原因和渠道类型排序：反爬登录态优先提示配置 Cookie，短正文优先走详情页/API/渲染兜底。
3. 每次补偿后更新质量画像，让管理端能看到补偿是否真正提升完整度。

## 三、事件分析升级

1. 事件摘要从“拼接短文本”升级为结构化分析快照。
2. 每个事件至少生成：一句话看懂、发生了什么、为什么重要、最新进展、为什么热、风险提示、来源对比。
3. 分析内容必须携带质量意识：来源少、详情不完整、疑似列表摘要时，要明确提示风险，不能包装成确定结论。

## 四、产品展示升级

1. `info-mvp` 事件详情页继续区分摘要和代表原文，后续增加“为什么热”“风险提示”“多平台观点”。
2. 管理后台质量报告展示每个渠道可用率、完整率、平均正文长度、主要失败原因、弱样本。
3. 运营动作聚焦“重抓低完整详情”“补 Cookie”“重建事件”“归档低质量内容”。

## 五、实施顺序

1. 第一批：统一质量画像、补偿队列接入、质量报告接入、事件分析新增热度原因和风险提示。
2. 第二批：按渠道重构详情抓取策略，逐个渠道做线上样本对比验收。
3. 第三批：升级事件聚合算法，引入实体、时间、语义和跨渠道证据。
4. 第四批：前端热点分析页升级，展示结构化事件分析结果。

## 当前批次验收标准

1. 同一条数据在补偿队列和质量报告中的完整度判断一致。
2. 文章型渠道短正文即使评分中等，也会进入补偿。
3. 事件详情 API 能返回“为什么热”和“风险提示”快照。
4. 质量报告弱样本能说明具体失败原因和建议动作。

## 2026-05-08 最新推进记录

1. 已补充 `AcquisitionQualityProfile` 统一质量画像，详情补偿、渠道质量报告、信息详情 API 都从同一套规则读取完整度、价值性、风险原因和建议动作。
2. 已把低质详情任务的 `strategy_hint` 从固定 `auto` 升级为质量画像推荐动作，例如 `retry_full_article_detail`、`search_secondary_detail_source`、`check_cookie_or_rendering_strategy`。
3. 已在详情补偿队列报告和后台页面展示补偿策略，运营可以按失败原因和建议动作判断下一步处理方式。
4. 已验证文章型渠道短正文、列表摘要、失败详情会进入补偿队列，并携带对应优先级和策略。
5. 已让详情补偿 worker 读取 `strategy_hint`。当任务目标是重抓完整文章或二跳详情时，partial 短正文不再被误判为补偿成功，会继续进入详情页 HTML 兜底或保留在队列中重试。
6. 已新增 `SecondarySearchDetailStrategy` 二跳搜索详情策略。对于 `search_secondary_detail_source` 任务，系统会按渠道域名搜索真实候选详情页，排除原热词页，再抓取候选正文进入质量管线。
7. 已为知乎、小红书、微博补充专用二跳搜索策略名称和查询域，后续可以继续替换为各渠道 API 或登录态渲染策略。
8. 已为 `check_cookie_or_rendering_strategy` 增加凭证诊断。缺少 `WEIBO_COOKIE`、`ZHIHU_COOKIE`、`XHS_COOKIE` 等必要凭证时，补偿任务会直接标记 `missing_required_credentials`，避免无效重试。
9. 已将事件构建升级为高质量来源优先：事件主标题、摘要、来源链接权重优先使用完整度和价值分更高的来源，时间线仍按发生时间展示。
10. 已在管理后台详情补偿总览增加“补偿策略趋势”，运营可以直接看到任务集中在完整文章重抓、二跳搜索还是 Cookie/渲染诊断。
11. 已在渠道质量报告增加治理优先级和治理建议，缺凭证渠道会明确提示需要配置的环境变量和重抓动作。
12. 已在事件构建中加入分析来源质量阈值，可用来源优先参与摘要生成，硬风险来源进入证据链风险区。
13. 已在事件详情 API 和 `info-mvp` 详情页增加证据链展示，包括可用来源、风险来源和平台来源分布。

## 下一批计划

1. 将微博、小红书、知乎的专用二跳搜索继续升级为“登录态 API + 渲染兜底”组合，减少搜索引擎依赖。
2. 后台增加针对缺失凭证的快捷筛选和重试按钮，缩短运营治理路径。
3. 对线上数据跑分渠道验收，按完整率、平均正文长度、可用率排序继续治理低分渠道。
4. 事件详情继续增强多平台观点差异，把不同渠道的立场和新增事实拆开展示。
5. 增加采集补偿效果追踪：补偿前后正文长度、完整度、价值分变化要可视化。
