# 微博渠道正文短问题调查与优化方案

时间：2026-05-02

## 现象

微博渠道当前能稳定采集热搜列表，但详情内容普遍只有几十字，评分偏低。线上轻量验证中：

- 主入口可采集 20 条热词。
- 多数详情来自 `hot_band_context`，长度约 64-81 字。
- `topic_search` 返回空。
- `mobile_search` 返回 432 或非 JSON。
- `s.weibo.com` 页面返回访客跳转。

## 当前真实入口状态

| 入口 | URL/策略 | 当前结果 | 是否能提供完整正文 |
| --- | --- | --- | --- |
| 热榜 API | `https://weibo.com/ajax/statuses/hot_band` | 200，可用 | 不能，只提供热词、热度、分类、标签 |
| 侧栏热搜 API | `https://weibo.com/ajax/side/hotSearch` | 200，可用 | 不能，只提供热词、热度 |
| 话题搜索 API | `https://weibo.com/ajax/search/topic?query=...` | `ok=-100`，要求登录 | 能，但当前未登录不可用 |
| 移动热搜 API | `https://m.weibo.cn/api/container/getIndex?...realtimehot` | 432 | 当前不可用 |
| 移动搜索 API | `https://m.weibo.cn/api/container/getIndex?...q=...` | 432 | 当前不可用 |
| 搜索网页 | `https://s.weibo.com/weibo?q=...` | 302 到 passport visitor | 当前不可用 |

## 根因分析

### 1. 当前保存的是“热搜词条”，不是微博正文

`hot_band` 的 `band_list` 字段包含：

- `word` / `note`
- `category`
- `rank` / `realpos`
- `num` / `raw_hot`
- `label_name`
- `transparency_tag`

这些字段适合做“热搜事件背景”，但不是原始微博帖子正文。即使抽取完整，也只能得到几十到一百字左右。

### 2. 详情正文入口需要登录态或更强浏览器上下文

真正能产生正文的入口是：

- 话题搜索结果中的 `statuses`
- 移动搜索结果中的 `mblog`
- 单条微博详情页
- 话题页/搜索页渲染后的卡片正文

但当前本地环境下：

- `ajax/search/topic` 返回 `ok=-100`，要求登录。
- `m.weibo.cn` 返回 432。
- `s.weibo.com` 返回访客跳转。

因此代码只能退回热榜上下文。

### 3. 当前质量评分和微博内容形态存在错位

微博热搜本身通常不是长文章。对微博渠道不能用“文章完整正文”的标准评价，需要区分：

- 热搜事件级详情：热词、分类、热度、上榜时间、媒体数量、讨论趋势。
- 微博帖子级详情：多条代表性微博正文、转发长文、评论/互动信息。

当前只有事件级详情，缺少帖子级详情，所以评分低是合理结果。

## 优化方案

### 方案 A：接入微博登录态，优先获取帖子正文

新增配置：

- `WEIBO_COOKIE`
- `WEIBO_XSRF_TOKEN` 或必要请求头
- `WEIBO_ENABLE_AUTH_DETAIL=true`

当前已先接入 `WEIBO_COOKIE`：

- 本地开发：放在项目根目录 `.env`，例如 `WEIBO_COOKIE=SUB=...; XSRF-TOKEN=...; _s_tentry=...`。
- 配置模板：`.env.example` 和 `info_aggregation/deploy.env.example` 已补充 `WEIBO_COOKIE=`。
- 容器部署：根目录 `docker-compose.yml` 和 `info_aggregation/docker-compose.prod.yml` 已注入 `WEIBO_COOKIE: ${WEIBO_COOKIE:-}`。
- 代码读取：`crawlers/weibo.py` 会按环境变量、当前目录 `.env`、父目录 `.env`、仓库根目录 `.env` 的顺序读取，并在微博热榜、话题搜索、移动搜索、网页兜底请求头中自动携带 Cookie。

采集链路：

1. 继续用 `hot_band` 获取热词。
2. 使用登录态请求 `ajax/search/topic` 或 `s.weibo.com/weibo?q=...`。
3. 从搜索结果中提取前 5-10 条代表性微博：
   - `text`
   - `longText.longTextContent`
   - `retweeted_status.text`
   - `retweeted_status.longText.longTextContent`
   - 作者、发布时间、互动数
4. 合并为一条事件详情：
   - 热搜背景
   - 代表性微博 1/2/3
   - 讨论热度和分类

优点：最符合目标，能得到真实正文。

风险：Cookie 失效、账号风控、请求频率限制。

### 方案 B：浏览器渲染搜索页，使用本地登录态

使用 Playwright/Chromium：

1. 打开 `https://s.weibo.com/weibo?q=关键词`。
2. 复用浏览器用户数据目录或注入 Cookie。
3. 等待搜索结果卡片渲染。
4. 提取正文卡片和 `展开` 后文本。

优点：更接近真实页面，能绕过部分接口签名问题。

风险：成本高、慢、需要登录态、容易触发风控。

### 方案 C：不追求帖子正文，把微博定义为热搜事件源

若短期不接入登录态，则明确微博只采集热搜事件：

1. 继续采集 `hot_band` / `sideHotSearch`。
2. 扩充事件上下文：
   - 排名、热度、分类、标签、上榜时间、媒体发布数量。
3. 调整微博质量 profile：
   - `complete_min_length` 从 80 降到 60。
   - value markers 增加 `热度值`、`热榜分类`、`当前排名`。
4. 后续事件分析阶段把微博作为“热度信号”，不作为正文主内容源。

优点：稳定，不依赖账号。

缺点：仍然没有真实微博正文，分析价值有限。

## 推荐路线

推荐采用 A + C 的双层策略：

1. 无 Cookie 时：微博作为热搜事件源，稳定提供热词、热度、分类、排名，质量评价按事件源标准计算。
2. 有 Cookie 时：升级为帖子级详情源，采集多条代表性微博正文，并把详情评分提高到 complete。
3. 浏览器渲染作为兜底，只用于管理页“立即采集”或低频补偿任务，避免定时任务成本过高。

## 需要改造的代码点

- `config.py`
  - 增加 `WEIBO_COOKIE`、`WEIBO_ENABLE_AUTH_DETAIL`、`WEIBO_DETAIL_MAX_POSTS`。
- `crawlers/weibo.py`
  - `_build_weibo_auth_headers()`：统一构造带 Cookie 的请求头。
  - `_fetch_topic_search()`：支持登录态，解析 `statuses`，记录失败原因。
  - `_fetch_search_page_cards()`：解析 `s.weibo.com` 卡片，作为 HTML 兜底。
  - `_compose_hot_event_context()`：统一拼接热榜上下文。
  - `resolve_detail()`：先取帖子正文，失败再退热榜上下文。
- `services/detail_pipeline.py`
  - 微博 profile 区分 `hot_event` 与 `post_detail`，或者降低事件型详情阈值。
- 测试
  - 无 Cookie：热榜上下文为 partial/complete event context。
  - 有 Cookie：搜索 statuses 合并为完整正文。
  - 登录跳转/432：记录失败原因，不误判为完整正文。

## 验收标准

### 无 Cookie 模式

- 热搜入口稳定返回 20 条。
- 详情长度达到 60-120 字。
- 状态至少为 partial。
- 不再把微博壳页面、登录页当详情。

### 有 Cookie 模式

- 每个热词至少提取 3 条代表性微博正文，或累计正文长度超过 300 字。
- `complete` 比例超过 70%。
- 平均详情分超过 80。
- 能识别长文和转发补充，不只保存标题。
