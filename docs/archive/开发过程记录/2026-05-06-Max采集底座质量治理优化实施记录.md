# Max 采集底座质量治理优化实施记录

日期：2026-05-06

## 目标

本轮优化围绕“采集到更多完整、有价值、可分析的数据”展开，不新增渠道，优先提升现有采集底座的可观测性、凭证治理、渠道质量验收和详情策略一致性。

## 优化计划

1. 统一微博、知乎、小红书等登录态凭证读取，避免每个爬虫重复解析 `.env`。
2. 暴露采集凭证健康状态，管理端可以判断低分是否由 Cookie 缺失引起。
3. 修正详情策略链分渠道评分，使文章类、社交类、财经类、体育类使用各自阈值。
4. 增加渠道级质量报告，按渠道统计真实数据完整率、可用率、失败原因、低分样本。
5. 增加本地真实样本验收工具，开发阶段可直接抽样检查线上入口和详情正文。

## 已完成改动

### 1. 凭证统一治理

新增 `services/credential_provider.py`：

- 统一从环境变量、当前目录 `.env`、服务目录 `.env`、仓库根目录 `.env` 读取凭证。
- 支持脱敏状态输出，不在日志或接口中泄露完整 Cookie。
- 内置渠道凭证要求：
  - 微博：`WEIBO_COOKIE`
  - 知乎：`ZHIHU_COOKIE`，可选 `ZHIHU_ZSE_93/ZHIHU_ZSE_96`
  - 小红书：`XHS_COOKIE`
- 微博、知乎、小红书爬虫已切换到统一凭证服务。

### 2. 详情策略链修正和可观测

更新 `services/detail_strategy_chain.py`：

- `run_detail_pipeline()` 现在会接收 `context.channel_code`，真正启用分渠道阈值。
- 策略执行新增 `execution_logs`，记录 strategy、status、score、content_length、failure_reason、elapsed_ms。
- 单个策略异常会转化为标准失败候选，不会直接中断整个详情链。

这项修正很关键：之前策略链里的 36氪、CSDN、Reuters 等文章渠道可能被默认短文本阈值误判为完整；现在会按渠道 Profile 判断，避免“短摘要冒充完整正文”。

### 3. 渠道级质量报告

新增 `services/channel_quality_report.py` 和接口：

- `GET /api/admin/channel-quality-report`
- `GET /api/admin/crawl-credential-report`

渠道质量报告会按渠道输出：

- real_count / seed_count，避免模拟数据污染真实质量。
- complete_ratio，真实完整正文比例。
- usable_ratio，完整或高分 partial 的可用比例。
- needs_attention_ratio，需要继续优化的低质量比例。
- avg_detail_score / avg_detail_content_length。
- top_failure_reasons / top_detail_strategies。
- weak_samples，最需要处理的低分样本。
- credential_health，关联 Cookie 是否配置。

### 4. 本地真实样本验收工具

新增 `tools/live_channel_quality_audit.py`：

```bash
cd info_aggregation
./.venv/bin/python tools/live_channel_quality_audit.py --channels weibo,zhihu,toutiao --limit 3
```

工具特性：

- 只抓线上样本，不写数据库。
- 每个渠道输出热点入口条数、样本标题、来源 URL、详情策略、详情状态、详情分数、正文长度、失败原因、正文摘要。
- 可用 `--skip-detail` 只验证热点入口。
- 适合每次优化某个渠道后做人工验收。

### 5. Reuters 线索恢复升级

更新 `crawlers/reuters.py`：

- Google News 兜底不再只保存 Google 跳转链接，会优先从摘要 HTML 中恢复 `https://www.reuters.com/...` 官方 URL。
- 如果恢复到官方 URL，详情阶段会先尝试 Reuters article API、JSON-LD、HTML 正文，再回退新闻索引摘要。
- 如果只能拿到 Google News 链接，仍明确走 `news_index_summary`，不伪装成全文。

这能把 Reuters 从“索引线索”尽量推进到“官方 URL + 可尝试全文”，同时保留失败时的可解释降级。

### 6. 小红书入口价值过滤

更新 `crawlers/xiaohongshu.py`：

- 探索页列表内容增加标签和互动指标。
- 过滤标题和正文都极短、没有标签、没有互动信号的低信息量笔记。
- 列表 `_search_content` 会带上更完整的入口上下文，减少详情失败时只剩标题。

小红书很多笔记天然短，如果入口没有互动和标签，进入事件分析的价值很低。本轮优化先把这类内容挡在入口层。

### 7. 微博 source_id 稳定性优化

更新 `crawlers/weibo.py`：

- 搜索 URL 使用 URL encode，避免中文热词在 URL 中不规范。
- 热榜 API 的 `source_id` 增加 rank 和热度值上下文，降低同一热词在不同阶段复用时被旧数据覆盖的概率。
- 后续还需要继续升级为话题 ID、mid 或代表性微博 ID，这是更彻底的方案。

## 产品层收益

1. 管理端能看见“哪个渠道低分、为什么低分、是不是 Cookie 缺失”。
2. 采集研发能用同一套验收工具验证真实线上数据，不再被 seed/mock 数据误导。
3. 详情评分更贴近渠道特点，减少标题和短摘要被误判为完整正文。
4. 微博、知乎、小红书这类强登录态渠道的配置路径更清晰，后续可继续接入 Cookie 过期检测。

## 后续继续优化方向

1. Reuters 仍需解决全文源能力，目前 sitemap 和 Google News 只能作为线索级内容。
2. 微博需要进一步把 source_id 从热榜上下文升级为话题 URL、mid 或代表性微博 ID。
3. 小红书需要扩展 topic/search 入口，并继续优化低信息量图文笔记过滤。
4. 知乎需要做 ZSE/Cookie 健康检测和文章/问题/回答类型拆分。
5. 后续管理端可以直接接入 `/api/admin/channel-quality-report`，把渠道质量报告展示出来。
