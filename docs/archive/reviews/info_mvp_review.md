# info-mvp 工程 Review 总结

## 1. 工程定位

**info-mvp** 是整个 info-ai 体系的**移动端前端**，基于 uni-app 构建，一套代码同时输出到 **H5** 和**微信小程序**。定位是 info-max 的移动版补充，让用户在微信生态和移动端浏览器中也能浏览热点事件。

核心页面：
- 首页：事件列表 + 搜索 + 筛选面板（分类+渠道+排序）
- 事件详情：时间线、多方视角、相关报道
- 登录页：JWT 认证
- 收藏夹与阅读历史

---

## 2. 系统架构

### 2.1 技术栈
- **Framework**: uni-app Vue3 + Vite
- **Cross-platform**: H5 + 微信小程序（MP-WEIXIN）
- **State Management**: Pinia
- **UI Components**: `@dcloudio/uni-ui`（uni 官方组件库）
- **Icons**: `uniicons` 字体图标
- **HTTP**: `uni.request` 封装
- **Language**: TypeScript

### 2.2 模块结构
```
info-mvp/src/
├── pages/              # 页面（uni-app 约定式路由）
│   ├── home/home.vue       # 首页
│   ├── event-detail/       # 事件详情
│   ├── info-detail/        # 内容详情
│   ├── login/login.vue     # 登录
│   ├── favorites/          # 收藏
│   └── history/            # 阅读历史
├── components/         # 组件
│   ├── EventList.vue
│   ├── FilterPanel.vue     # 折叠式筛选面板
│   ├── FilterBar.vue       # 搜索框
│   ├── EventCategoryTabs.vue
│   └── SkeletonBlock.vue
├── composables/        # 组合式逻辑
│   ├── useAuth.ts
│   ├── useFavorite.ts
│   ├── useFilterMemory.ts
│   └── useListLoad.ts
├── stores/             # Pinia 状态
│   └── user.ts
├── services/           # API 封装
│   ├── api.ts
│   └── request.ts
├── types/              # 类型定义
└── App.vue             # 全局样式（CSS 变量设计系统）
```

### 2.3 架构评价
**优点**：
- **Pinia 状态管理**：用户认证状态集中管理，跨页面同步
- **composables 复用**：`useListLoad`、`useFavorite` 等逻辑抽象到位
- **CSS 变量设计系统**：`App.vue` 中定义了完整的色彩/间距/阴影变量，跨页面一致性高
- **条件编译**：`#ifdef MP-WEIXIN` 处理分享、登录等平台差异
- **筛选面板交互设计佳**：折叠/展开动画流畅， pill 按钮设计现代

**问题**：
- **跨平台兼容性隐患**：uni-app 的 `view`/`text` 组件在 H5 和微信小程序表现有差异，部分样式需平台补丁
- **无虚拟列表**：事件列表长滚动时 DOM 节点累积，低端机卡顿
- **请求拦截器逻辑复杂**：401 处理、静默刷新、token 注入耦合在一起，易出 race condition
- **首页导航栏问题**：`pages.json` 和 `manifest.json` 的标题配置混乱，曾出现顶部双标题

## 3. 代码逻辑

### 3.1 首页 (`home/home.vue`)
- 筛选条件持久化：本地缓存 + 服务端同步（登录后）
- 下拉刷新 + 上拉加载更多，符合移动端交互习惯
- 返回顶部按钮（带箭头图标）
- **近期优化**：去掉系统导航栏（`navigationStyle: custom`）、去掉品牌标题区域、筛选面板新增渠道选项

### 3.2 筛选面板 (`FilterPanel.vue`)
- 折叠态显示"分类 · 渠道 · 排序"，点击展开为白色卡片
- pill 按钮选中态设计精致（蓝色背景 + 阴影）
- 但渠道数据来源为 `getChannels()` API，首次加载前显示"全部渠道"

### 3.3 事件列表 (`EventList.vue`)
- 卡片式布局，左侧彩色条带标识分类
- 热度分、来源数、渠道 badge 展示信息丰富
- **问题**：卡片内 `one_line_summary` 在小屏手机上可能换行过多

### 3.4 登录与认证
- `useAuth` composable 封装了登录态检查
- `request.ts` 拦截器中处理 401 和 token 刷新
- **问题**：未登录用户访问收藏/历史时，未做前置拦截，而是等 API 返回 401 后再跳转

## 4. 页面布局与视觉

### 4.1 布局特点
- 全屏移动端布局，顶部自定义导航栏（已去除系统导航栏）
- 搜索框 + 筛选条件 + 事件列表三段式结构
- 底部安全区适配（uni-app 自动处理）

### 4.2 色彩系统
- 与 info-max 保持一致：蓝色主色调 + 浅灰背景 + 白色卡片
- CSS 变量在 `App.vue` 的 `page` 选择器中定义，uni-app 全页面生效
- 分类色带：蓝、紫、青、橙、玫红（与 info-max 统一）

### 4.3 近期 UI 优化轨迹
- 减少黑色占比（背景改为 `#f7f8fa`）
- 筛选面板改为自动隐藏式，收起后仅显示当前选中项
- 字体缩小为 `24rpx`，颜色改为 `#595959` 浅黑
- 按钮、标签大小重新调整，更加精致

### 4.4 UI 问题
- **首页顶部区域仍显拥挤**：去除品牌标题后，搜索框 + 筛选条件堆叠，首屏可见内容少
- **渠道筛选 pill 过多时换行**：渠道数量多（10+）时，展开面板高度过高
- **微信小程序原生导航栏去除后**：顶部状态栏（电量、时间）可能遮挡自定义导航内容
- **深色模式缺失**：夜间阅读体验差

## 5. 产品定位分析

info-mvp 的定位是**"移动端轻量入口"**，主攻微信小程序生态和 H5 分享传播。

**当前匹配度**：
- 微信小程序：匹配微信用户习惯，分享路径短
- H5：便于短信/社群传播
- 轻量阅读：列表 → 详情流程顺畅

**与 info-max 的差异化**：
- info-max 是 PC 主站（深度阅读、大屏浏览）
- info-mvp 是移动补充（碎片时间、微信生态）

**缺失的产品能力**：
- 无小程序订阅消息（无法推送热点）
- 无微信好友分享卡片定制（分享时仅为默认标题）
- 无埋点（无法分析小程序用户行为）
- 无离线缓存（弱网环境下体验差）

## 6. 问题汇总与优化方向

### 6.1 高优先级（近期）

| 问题 | 影响 | 优化方向 |
|------|------|----------|
| 首页首屏内容少 | 用户一屏只能看到 1-2 条事件 | 进一步压缩顶部区域，或搜索框悬浮置顶 |
| 渠道 pill 过多 | 展开面板过长，操作不便 | 渠道改为横向滚动或二级展开 |
| 未登录拦截后置 | 用户先点收藏再跳转，体验差 | 前置拦截：点击收藏/历史时未登录直接弹登录框 |
| 微信小程序顶部适配 | 状态栏可能遮挡内容 | 动态计算 `statusBarHeight`，留出安全距离 |
| 虚拟列表缺失 | 长列表低端机卡顿 | 引入 `uni-list` 虚拟滚动或分页加载替代无限滚动 |

### 6.2 中优先级（中期）

| 问题 | 影响 | 优化方向 |
|------|------|----------|
| 无小程序订阅消息 | 用户回流手段缺失 | 接入微信订阅消息，热点事件推送 |
| 分享卡片无定制 | 分享转化率低 | 自定义 `onShareAppMessage` 分享标题/图片 |
| 无离线缓存 | 弱网/无网无法浏览 | 使用 `uni.setStorage` 缓存事件列表 |
| 无埋点分析 | 无法量化用户行为 | 接入微信小程序数据分析或自建埋点 |
| H5 与小程序体验差异 | 维护两套交互逻辑成本高 | 统一交互设计，减少 `#ifdef` 分支 |

### 6.3 低优先级（长期）

| 问题 | 影响 | 优化方向 |
|------|------|----------|
| 无暗黑模式 | 夜间体验差 | CSS 变量支持暗色主题 |
| 无骨架屏细节优化 | 首屏感知 loading 时间长 | 针对图片/文字分别做骨架动画 |
| 多端输出未验证 | 支付宝/百度小程序未测试 | 明确产品边界，放弃低优先级平台 |
| 无 PWA 能力（H5） | 无法添加到桌面 | H5 模式接入 Service Worker |

---

## 7. 总体评分

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构设计 | 7/10 | Pinia + composables 设计好，但跨平台兼容性需关注 |
| 代码质量 | 6.5/10 | 逻辑清晰，但请求拦截器复杂、401 处理有 race condition 风险 |
| UI/UX 设计 | 6.5/10 | 近期优化明显，但首屏拥挤、渠道展开过长 |
| 性能优化 | 5.5/10 | 无虚拟列表、无图片懒加载、无缓存策略 |
| 产品匹配 | 6.5/10 | 微信生态匹配好，但缺少推送、分享定制、埋点 |

**一句话总结**：info-mvp 是一个"迭代快速、交互精致但性能和产品化有待加强"的 uni-app 移动前端，近期应重点解决**首屏空间利用率**和**渠道筛选交互**，中期补齐**订阅推送**和**分享定制**能力。
