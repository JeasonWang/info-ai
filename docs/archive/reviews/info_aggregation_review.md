# info_aggregation 工程 Review 总结

## 1. 工程定位

**info_aggregation** 是整个 info-ai 体系的数据采集层（Data Acquisition Layer），负责从多源渠道（微博、头条、小红书、36氪、东方财富等）爬取原始信息，经过清洗、详情抓取后写入 MySQL，为上游的 info-serve 提供数据底座。

核心职责：
- 多渠道爬虫调度与管理
- 列表页 + 详情页两级抓取
- 内容清洗与去重
- 详情补偿任务（低质量/失败内容的二次抓取）
- 数据质量监控（重复、缺失、低分）

---

## 2. 系统架构

### 2.1 技术栈
- **Language**: Python 3
- **Web Framework**: FastAPI（提供管理 API）
- **ORM**: SQLAlchemy 2.0
- **Scheduler**: APScheduler（定时任务调度）
- **HTTP Client**: requests + Playwright（详情页渲染）
- **Database**: MySQL（生产）/ SQLite（测试）
- **Content Extraction**: trafilatura + 自定义 HTML 正则

### 2.2 模块结构
```
info_aggregation/
├── crawlers/           # 渠道爬虫实现（12+ 渠道）
│   ├── base.py         # 爬虫抽象基类
│   ├── registry.py     # 爬虫注册中心
│   └── *.py            # 各渠道实现
├── cleaners/           # 内容清洗（当前为空）
├── database/           # SQLAlchemy 模型与会话
│   ├── models.py       # 全量数据模型
│   └── session.py      # DB 连接管理
├── services/           # 业务逻辑（详情 pipeline、质量监控）
├── config.py           # 全局配置
└── tests/              # 单元测试
```

### 2.3 架构评价
**优点**：
- 爬虫基类设计规范，继承+注册机制便于扩展新渠道
- 数据库模型完整，覆盖渠道、分类、内容、事件、时间线、摘要等多维度
- 详情抓取 Pipeline 设计合理，支持策略回退和质量评分
- 采集健康度、数据质量快照等可观测性设计前置

**问题**：
- **缺少服务层拆分**：FastAPI 路由、业务逻辑、数据访问耦合在单一层级
- **无消息队列**：爬虫调度纯内存化（APScheduler），重启丢状态，无法水平扩展
- **Playwright 依赖过重**：详情抓取引入浏览器渲染，资源消耗大，未做资源池管理
- **配置硬编码**：渠道列表、分类枚举写在 `config.py` 中，无法热更新

## 3. 代码逻辑

### 3.1 爬虫基类 (`crawlers/base.py`)
- 封装了请求重试、UA 轮换、异常捕获等通用能力
- `safe_crawl()` / `safe_fetch_detail()` 提供了防御式编程范式
- 详情页文本提取使用正则匹配 HTML，鲁棒性不足（目标站点改版即失效）

### 3.2 数据库模型 (`database/models.py`)
- 模型设计较为专业：索引策略合理（`idx_event_category_score`、`idx_detail_job_status_priority`）
- 唯一约束覆盖去重场景（`uq_source_channel`、`uq_event_item`）
- 使用 `lazy="joined"` 较多，需警惕 N+1 问题在复杂查询时爆发

### 3.3 测试
- 测试文件数量尚可（10+），但覆盖以单渠道爬虫为主
- 缺少集成测试（pipeline 端到端、调度器行为）

### 3.4 代码质量红旗
- `cleaners/` 目录为空，清洗逻辑未落地
- 部分爬虫直接解析 JSON API，部分解析 HTML，未统一抽象
- `requests.Session` 未做连接池复用优化
- 日志仅输出到文件，未接入结构化日志或 tracing

---

## 4. 产品定位与业务价值

info_aggregation 是整套系统的**数据生命线**。没有持续、稳定、高质量的数据采集，上游的事件聚合、用户推荐都无从谈起。

当前定位偏重"技术实现"，产品化能力不足：
- 无法动态增删渠道（需改代码+部署）
- 采集策略调整无 A/B 能力
- 反爬对抗手段单一（仅 UA 轮换 + 简单延时）

---

## 5. 问题汇总与优化方向

### 5.1 高优先级（近期）

| 问题 | 影响 | 优化方向 |
|------|------|----------|
| 爬虫调度无持久化 | 服务重启丢任务、无法扩容 | 引入 Redis/RabbitMQ 作为任务队列，APScheduler 仅作触发器 |
| Playwright 资源消耗 | 详情抓取成本高、易 OOM | 优先使用 trafilatura/文本提取，Playwright 作为 fallback；引入资源池 |
| 反爬能力弱 | 高频采集易被封 | 增加代理池、请求指纹随机化、分布式调度 |
| 配置硬编码 | 新增渠道需发版 | 渠道配置化入库，管理后台动态下发 |

### 5.2 中优先级（中期）

| 问题 | 影响 | 优化方向 |
|------|------|----------|
| cleaners 目录为空 | 脏数据流入下游 | 落地清洗规则（HTML 标签过滤、广告去除、敏感词检测） |
| 详情页提取鲁棒性差 | 站点改版导致大规模失败 | 引入 Readability/trafilatura 为主，正则为辅 |
| 无数据血缘追踪 | 难以定位质量问题根因 | 增加内容版本链、采集溯源日志 |
| 测试覆盖不足 | 重构风险高 | 补充 pipeline 集成测试、mock 外部站点 |

### 5.3 低优先级（长期）

| 问题 | 影响 | 优化方向 |
|------|------|----------|
| 单体架构瓶颈 | 渠道增多后无法横向扩展 | 拆分为"调度中心 + 采集 Worker"微服务架构 |
| 日志非结构化 | 排查效率低 | 接入 JSON 结构化日志 + ELK/Loki |
| 缺少实时流能力 | 热点发现延迟高 | 引入 Kafka + Flink 做实时聚合 |

---

## 6. 总体评分

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构设计 | 6/10 | 分层清晰，但缺少持久化队列和扩展性设计 |
| 代码质量 | 6.5/10 | 基类设计好，但清洗层缺失、部分硬编码 |
| 可维护性 | 5.5/10 | 渠道增多后维护成本高，配置化不足 |
| 可观测性 | 7/10 | 健康度、质量快照设计前置，值得肯定 |
| 业务匹配 | 7/10 | 核心能力具备，产品化、动态化不足 |

**一句话总结**：info_aggregation 是一个"能跑通、难扩展"的采集系统，当前阶段应优先解决**调度持久化**和**详情抓取成本**两个问题，否则渠道扩容将成为瓶颈。
