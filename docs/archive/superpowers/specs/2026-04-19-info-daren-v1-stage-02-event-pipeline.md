# 信息达人 V1 阶段文档 02：事件理解流水线

## 文档信息

- 项目名称：信息达人
- 阶段主题：事件理解流水线设计
- 当前状态：已在对话中确认
- 更新时间：2026-04-19

## 1. 设计目标

`信息达人 V1` 的关键不只是采集信息，而是把采集到的多源内容转化为用户可以直接消费的“事件”。

这意味着系统需要一条稳定、可扩展、可替换的处理链，把原始内容逐步提升为：

- 可聚合的内容单元
- 可理解的事件对象
- 可展示的事件摘要与时间线
- 可排序的事件流

## 2. 总体处理链

建议 V1 的核心处理顺序如下：

`raw_item → normalized_item → 候选聚类 → 事件合并/新建 → 摘要/时间线生成 → 评分发布`

这条链路既能支持尽快落地 MVP，也能为后续版本扩展出更强的理解能力打下基础。

## 3. 处理阶段设计

### 3.1 采集入库

目标：

- 将爬虫或外部 API 获取到的原始内容保存在 `raw_item`
- 保证原始来源可追溯
- 为后续重算和纠错保留基础数据

原则：

- 采集层尽量少做产品判断
- 先保住原始内容与元信息
- 不把复杂智能逻辑写死在采集器里

## 3.2 标准化与去重

目标：

- 把不同来源内容统一成 `normalized_item`
- 提取后续理解处理会用到的统一字段

统一字段建议包括：

- 标题
- 摘要
- 正文片段
- 发布时间
- 来源渠道
- 分类
- 主体/实体候选
- 标签候选
- 原始 URL

同时在这一层执行基础去重，避免同一内容反复流入理解层。

## 3.3 事件候选识别

目标：

- 先把明显属于同一事件的内容初步拉到一起
- 形成候选事件簇

V1 建议优先使用以下信号：

- 时间窗口接近
- 标题关键词重合
- 命名实体重合
- 分类一致或接近

这一层的重点是先建立“足够可用”的事件候选，不要求一开始就做到非常复杂的精准聚类。

## 3.4 事件合并 / 新建

目标：

- 判断候选内容应归入已有事件，还是新建一个事件

这一步建议单独做成处理器，原因是它将来最可能持续升级：

- 可以先用规则
- 后续可以引入 embedding 相似度
- 再后续可以接人工校验或运营修正

这一步的输出是正式的 `event` 与 `event_item_link` 关系。

## 3.5 事件理解生成

在事件级别生成用户真正看到的内容。

V1 建议至少生成以下几类摘要：

- `one_line`：一句话看懂
- `what_happened`：发生了什么
- `why_it_matters`：为什么重要
- `latest_update`：最新进展

建议这些内容通过 `event_summary_snapshot` 保存，而不是直接覆盖事件主表字段，方便：

- 后续升级生成策略
- 保留版本
- 支持重算
- 支持人工校正

## 3.6 时间线提取

目标：

- 从事件关联内容里抽取关键节点
- 形成按时间排序的 `event_timeline_entry`

V1 建议重点保证：

- 时间顺序清晰
- 节点数量适中
- 描述足够简短
- 节点可回溯到原始来源

V1 不需要一次性追求复杂的事件抽取系统，先把“用户能看懂发展过程”做好。

## 3.7 评分与发布

目标：

- 给每个事件生成首页排序所需的综合分
- 把事件发布到事件流 API

V1 建议至少包括以下分数：

- `heat_score`：热度分
- `freshness_score`：时效分
- `source_diversity_score`：来源多样性分
- `composite_score`：综合排序分

首页排序基于 `composite_score`，但保留分数拆解字段，方便后续调权重。

## 4. 可扩展性原则

这条流水线必须从一开始就按“处理链”设计，而不是堆成一个大函数。

建议每个阶段都是独立处理器，例如：

- normalize processor
- dedupe processor
- cluster processor
- event merge processor
- summary processor
- timeline processor
- score processor

这样后续演进时可以单独替换某一段，而不会冲击整条链。

## 5. V1 到后续版本的演进空间

### 5.1 聚类能力

- V1：规则聚类
- V2：规则 + 语义相似度
- V3：引入人工校正和反馈学习

### 5.2 摘要能力

- V1：模板化 / 轻模型摘要
- V2：更自然的一句话和背景补充
- V3：不同人群差异化摘要

### 5.3 时间线能力

- V1：从已知时间信息中整理节点
- V2：从正文中抽取更细粒度事件节点
- V3：支持事件状态变化、里程碑和长期跟踪

### 5.4 排序能力

- V1：热度 + 时效 + 来源数量
- V2：加入来源可信度和事件阶段
- V3：加入用户兴趣与个性化调权

## 6. 与 MVP 的关系

虽然完整方向是“全面的事件理解产品”，但这条流水线允许先做一个重点明确的 MVP：

- 先打通最短闭环
- 先支持少量重点来源
- 先做规则版聚类与摘要
- 先让用户能刷到“事件流”并点进“时间线详情”

后续可以在不改变总体架构的前提下，逐步增强每一个处理环节。

## 7. 当前阶段结论

这一阶段已经明确：

- `信息达人` 的核心不是“采集很多内容”，而是“把内容转成事件”
- 事件理解层是整套产品的核心引擎
- 处理链必须可插拔、可演进、可重算
- MVP 可以从简，但这条链路必须为全面产品预留扩展空间
