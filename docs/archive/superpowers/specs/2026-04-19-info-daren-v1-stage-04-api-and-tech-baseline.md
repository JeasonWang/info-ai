## 1. 当前技术栈约束

当前产品设计明确使用以下技术栈：

- `Python`
- `Go`
- `MySQL`
- `Redis`
- `MQ`
- `Vue3`
- `uni-app`

其中数据库已明确选择：

- `MySQL`

## 2. 架构设计目标

在上述技术栈下，架构设计需要同时满足两个目标：

1. 尽快交付第一个可运行的 MVP
2. 为后续成长为全面产品保留清晰扩展空间

因此，架构不应追求首版“技术堆栈全上”，而应追求：

- 组件职责清晰
- 系统边界清晰
- 后续可以自然扩展
- 首版可以快速落地

## 3. MVP API 设计

MVP 建议先围绕 3 个核心接口展开。

### 3.1 首页事件流 API

接口建议：

`GET /api/events`

用途：

- 首页综合流
- 分类频道流
- 分页加载
- 基于 `热度 + 时效` 的默认排序

请求参数建议：

- `category_code`
- `page`
- `page_size`
- `sort`

其中：

- `category_code` 默认值建议为 `all`
- `sort` MVP 默认值建议为 `composite`

返回字段建议：

- `id`
- `title`
- `one_line_summary`
- `primary_category`
- `heat_score`
- `freshness_score`
- `composite_score`
- `last_updated_at`
- `source_count`
- `source_badges`
- `new_update_count`

设计原则：

- 首页接口只返回“刷流必需字段”
- 不在首页接口中塞入详情页所有数据

### 3.2 事件详情 API

接口建议：

`GET /api/events/{event_id}`

用途：

- 事件详情页首屏
- 时间线展示
- 多来源归纳
- 背景补充
- 原始来源跳转

返回结构建议包括：

#### event

- 标题
- 一句话摘要
- 主分类
- 热度
- 最后更新时间

#### timeline

- 时间线节点数组

#### summaries

- `what_happened`
- `why_it_matters`
- `latest_update`

#### source_views

- 多平台归纳后的观点摘要

#### representative_sources

- 代表性原始来源列表

设计原则：

- 首页负责“刷”
- 详情负责“看懂”

### 3.3 分类入口 API

接口建议：

`GET /api/event-categories`

用途：

- 首页顶部分类入口
- 后续频道页复用

返回字段建议：

- `name`
- `code`
- `icon`
- `display_order`

## 5. 技术架构基线

在现阶段，推荐按职责划分技术栈，而不是机械平均分配。

### 5.1 Python

建议作为首阶段核心后端语言，负责：

- 爬虫采集
- 数据清洗与标准化
- 事件聚类
- 摘要生成
- 时间线整理
- MVP 事件 API

原因：

- 你现有 `info_aggregation` 已经是 Python
- Python 在爬虫、数据处理、文本理解上更高效
- 有利于 MVP 快速演进

### 5.2 Go

建议作为后续高性能服务语言，优先用于：

- 高并发网关
- 实时聚合服务
- 高吞吐任务处理器
- 后续需要极致性能的独立服务

原则：

- Go 现在先纳入整体架构规划
- MVP 阶段不强制拆出 Go 服务
- 避免为了“技术完整”而过早引入跨语言微服务复杂度

### 5.3 MySQL

作为核心业务数据库，负责：

- 原始内容元数据
- 标准化内容
- 事件主表
- 事件时间线
- 事件摘要快照
- 分类与渠道配置

原则：

- 以 MySQL 作为权威存储
- 所有核心业务对象必须可在 MySQL 中追溯

### 5.4 Redis

建议作为高频辅助存储，优先用于：

- 热门事件缓存
- 首页分页缓存
- 分布式锁
- 去重辅助
- 任务状态暂存

原则：

- Redis 是加速层，不是最终业务真相来源

### 5.5 MQ

建议作为异步流水线组件，优先用于：

- 采集完成后的标准化任务分发
- 聚类任务触发
- 摘要与时间线生成任务触发
- 后续事件重算

原则：

- 先把异步处理链设计清楚
- 首版实现可轻量
- 后续根据规模决定具体 MQ 产品和部署方式

当前确认的执行策略：

- 先把 MQ 在系统中的位置设计清楚
- MVP 不强制引入重型消息系统
- 首版可以先用轻量异步方式过渡，但接口与处理链边界要按未来 MQ 模式来设计

### 5.6 Vue3

建议作为首版 Web / H5 主前端技术，优先用于：

- 信息达人 Web 端
- 首版移动浏览器适配
- 管理端页面

### 5.7 uni-app

建议作为后续小程序 / 多端承载方案，优先用于：

- 微信小程序
- 多端轻应用扩展

原则：

- MVP 先把 Web / H5 打磨好
- uni-app 在第二阶段以后再承接更多端扩展

## 6. 当前阶段的推荐实施策略

在不改变长期方向的前提下，当前最推荐的实施方式是：

### 6.1 MVP 阶段

- 后端主力：`Python + MySQL + Redis`
- 异步处理：预留 `MQ` 设计接口
- 前端主力：`Vue3`

### 6.2 增长阶段

- 引入更完整的 `MQ`
- 根据吞吐与实时性需求拆分 Go 服务
- 推进 `uni-app` 端

## 7. 当前阶段结论

这一阶段明确了两件事：

1. `MVP API` 已经具备清晰边界
2. 技术栈分工应围绕“快速落地 + 长期扩展”来设计，而不是一开始就把所有技术都重度引入
