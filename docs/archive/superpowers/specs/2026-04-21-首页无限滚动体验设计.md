# 首页无限滚动体验设计

## 背景

首页移动端已经改为“信息流优先 + 粘性频道”。继续保留底部分页页码会打断浏览节奏，尤其在手机上需要用户主动找页码、点击下一页，不符合“简洁、快速、有价值”的体验定位。

本轮优化将底部分页替换为向下滚动自动加载。

## 目标

- 移除首页底部页码分页。
- 用户接近列表底部时自动加载下一页。
- 新数据追加到当前列表末尾，不替换已有内容。
- 切换频道或搜索时重置列表，从第一页重新加载。
- 底部只展示轻量状态，不干扰信息流阅读。

## 交互规则

### 初次加载

首页进入时加载第 1 页事件。

### 自动加载更多

页面底部放置一个不可见或轻量的“加载哨兵”区域。

当哨兵进入视口时：

- 如果正在加载，不发起重复请求。
- 如果已加载数量达到 `total`，不继续请求。
- 如果还有下一页，自动请求 `page + 1`。
- 请求成功后，把新事件追加到当前列表末尾。

### 切换频道

用户切换频道时：

- 清空旧列表。
- 将页码重置为 1。
- 请求新频道第 1 页。
- 页面平滑回到顶部。

### 搜索

用户提交搜索时：

- 应用关键词。
- 清空旧列表。
- 将页码重置为 1。
- 请求搜索结果第 1 页。

### 底部状态

底部不显示页码，只显示轻量状态：

- 正在加载更多热点...
- 已经看完当前频道
- 加载失败，提供重新加载按钮

如果还有更多内容但尚未触发加载，不展示额外提示，避免打扰阅读。

## 技术方案

使用 `IntersectionObserver` 监听底部哨兵元素。

状态集中在 `HomeView.vue`：

- `loading`：首屏加载状态。
- `loadingMore`：加载更多状态。
- `eventPage`：保留后端分页元信息。
- `items`：继续沿用 `eventPage.items`，追加时更新为旧列表加新列表。
- `hasMore`：由 `eventPage.items.length < eventPage.total` 计算得到。

`EventList.vue` 只负责展示：

- 事件列表。
- 首屏加载骨架屏。
- 空状态。
- 底部加载状态和重试入口。

不再依赖 `PaginationBar`。

## 非目标

- 不修改后端分页接口。
- 不引入虚拟列表。
- 不新增复杂筛选器。
- 不改变详情页。

## 验收标准

- 首页不再显示分页页码、上一页、下一页。
- 滚动到底部附近会自动请求下一页。
- 下一页数据追加在已有事件后面。
- 切换频道和搜索会重置列表，不混入旧频道或旧关键词结果。
- 已加载完全部数据后，底部显示“已经看完当前频道”。
- 前端测试和生产构建通过。

## 风险与处理

- 风险：重复触发导致连续请求。
- 处理：使用 `loadingMore` 和 `hasMore` 双重判断。

- 风险：浏览器测试环境没有 `IntersectionObserver`。
- 处理：测试中使用 mock observer，显式触发交叉回调。

- 风险：自动加载失败后用户不知道如何恢复。
- 处理：底部展示轻量错误和重新加载按钮。
