# 信息达人 V1 阶段文档 03：MVP 范围与版本路线

## 文档信息

- 项目名称：信息达人
- 阶段主题：MVP 范围切分、最小可运行闭环、版本路线
- 当前状态：已在对话中确认整体方向
- 更新时间：2026-04-19

## 1. MVP 目标

在不改变长期产品方向的前提下，`信息达人` 的第一个可运行版本应尽快完成一个最短闭环：

- 用户打开应用后能看到 `事件流`
- 每个事件能显示 `一句话看懂`
- 用户点进详情后能看到 `事件时间线`
- 事件下方能看到 `多来源归纳` 与 `代表性原始来源`

MVP 的目标不是做全，而是先证明这件事情有产品价值：

`大众用户愿意用“事件流 + 一句话摘要 + 时间线”替代多平台来回切换。`

## 2. MVP 产品形态

建议采用 `极简闭环型` 路线。

### 2.1 MVP 首页

首页默认展示：

- 全网综合事件流
- 顶部分类切换入口
- 事件卡片

每张事件卡片至少包含：

- 事件标题
- 一句话看懂
- 热度 / 更新时间
- 来源平台标签
- 最新进展数量

### 2.2 MVP 详情页

详情页至少包含：

- 事件标题
- 一句话看懂
- 事件时间线
- 多平台归纳
- 背景补充
- 代表性来源列表

### 2.3 MVP 管理能力

为了让首版能更快跑通，建议先只做轻量管理能力：

- 手动触发采集
- 查看渠道状态
- 查看事件生成结果

复杂运营后台可以后置。

## 3. MVP 必做功能

### 3.1 数据采集

- 先接入少量但有代表性的来源
- 支持定时更新
- 支持手动触发一次抓取

### 3.2 事件聚类

- 基于规则完成初版事件聚合
- 能把相同热点的多源内容并到同一事件下

### 3.3 一句话摘要

- 为每个事件生成一条适合首页阅读的短摘要
- 长度要明显短于普通正文描述

### 3.4 时间线生成

- 为详情页输出 3 到 8 个关键节点
- 节点要尽量按时间顺序且可追溯来源

### 3.5 事件流接口

- 支持综合流
- 支持分类筛选
- 支持分页
- 支持热度 + 时效混合排序

### 3.6 事件详情接口

- 返回事件主信息
- 返回时间线
- 返回来源摘要
- 返回背景补充
- 返回原始来源列表

### 3.7 前端展示

- 首页事件流
- 分类切换
- 详情页
- 基础加载态 / 空态 / 异常态

## 4. MVP 暂不做

为保证尽快交付首个可运行版本，以下能力不进入 MVP：

- 个性化推荐
- 用户登录体系
- 评论互动
- 收藏 / 历史 / 提醒体系的完整重构
- 复杂搜索
- 人工审核工作台
- 多终端一次性全部适配
- 复杂可视化报表

## 5. MVP 推荐来源范围

建议首版只接入少量高价值来源，形成“综合热点 + 垂类补充”的基本盘。

建议优先来源：

- 微博
- 今日头条或同类新闻聚合源
- 路透
- 36氪
- 东方财富

后续再逐步加入：

- 知乎
- 小红书
- 掘金
- CSDN
- 体育来源
- 更多国际新闻与财经来源

这样做的好处是：

- 先快速建立跨平台事件聚合的基础能力
- 控制数据接入复杂度
- 降低首版维护成本

## 6. MVP 成功标准

首个可运行版本建议以“能用”和“能验证价值”为核心，而不是追求全面覆盖。

建议成功标准包括：

- 用户能在首页连续刷到聚合后的事件，而不是原始信息碎片
- 大部分重点事件都有一句话看懂
- 事件详情能输出可读时间线
- 首页排序基本符合“热度 + 时效”
- 主要来源更新后，事件能在可接受时间内刷新

## 7. 版本路线图

### 7.1 M0：技术改造准备

目标：

- 保留当前 `info_aggregation` 和 `info-max`
- 新增事件模型与事件 API 的骨架
- 不影响当前 demo 可运行性

### 7.2 M1：首个可运行 MVP

目标：

- 打通 `采集 → 标准化 → 聚类 → 事件 → 前端展示`
- 首页可刷事件流
- 详情可看时间线
- 形成一个完整的产品闭环

### 7.3 M2：体验增强版

目标：

- 优化聚类质量
- 优化一句话摘要质量
- 优化时间线可读性
- 增加更多来源与频道

### 7.4 M3：平台化扩展版

目标：

- 搜索
- 订阅
- 推荐
- 人工运营
- 事件标签体系
- 更强的多端能力

## 8. MVP 实施原则

### 8.1 先闭环，再扩功能

先保证用户完整走通：

`打开首页 → 看到事件 → 点进详情 → 看懂时间线`

### 8.2 先规则，再智能升级

V1 不要被“必须先做很强的 AI 理解”卡住。

建议首版：

- 先用规则聚类
- 先用规则/模板摘要
- 先用基础时间线抽取

后面再逐步增强模型能力。

### 8.3 先少量来源，再扩覆盖

先选最有代表性的 5 个左右来源跑通闭环，再扩平台。

### 8.4 先事件 API，再重构前端

前端改造应围绕新的事件 API 展开，而不是继续直接消费旧 `info` 列表。

## 9. 当前阶段结论

这一阶段明确了：

- 第一版采用 `极简闭环型`
- 目标是尽快做出第一个可运行产品，而不是做完整平台
- MVP 只保留最能体现价值的能力
- 长期产品能力通过阶段演进逐步扩展

## 10. 下一阶段待讨论主题

下一阶段建议进入：

- MVP 首页 API 设计
- MVP 详情 API 设计
- 事件表与关联表的最小数据库方案
- 基于现有两个 demo 的迁移实施路径
